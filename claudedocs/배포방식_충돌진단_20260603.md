# AIMAX 배포 방식 충돌 진단 리포트 (A단계)

작성일: 2026-06-03
범위: 배포/패키징 방식의 충돌·중복·죽은코드 진단. **코드 변경 없음, 분석 전용.**
목적: "환경 차이 오류" 스트레스의 구조적 원인 규명 → 무엇을 버리고 무엇을 남길지 결정.

---

## 0. 한 줄 결론

문제는 부품이 모자란 게 아니라 **배포 경로가 곱셈으로 폭발**해서다.
서로 다른 5개 빌드 도구 × 2개 코드베이스 포크 × 5개 브랜드 × 2개 플랫폼 × 2개 패키지형태가
모두 살아있어서, 오류가 날 표면적이 수십 배로 늘어나 있다.

---

## 1. 현재 배포 매트릭스 (곱셈 폭발 실측)

| 축 | 현재 갖고 있는 것 | 개수 |
|---|---|---|
| 얼리는 도구(freeze) | PyInstaller(`build.py`) + Nuitka(`build_nuitka.py`) | 2 |
| 코드베이스 | 메인 `app.py`(293KB) + `split_version/app.py`(282KB) 복제 포크 | 2 |
| 브랜드 | AIMAX / 예리 / 현주 / Office-Manager(지은) / Pencil(나경) | 5 |
| 플랫폼 | macOS / Windows | 2 |
| 패키지 형태 | Setup.exe(설치형) / portable.exe / .dmg | 2~3 |
| 추가 소스 경로 | `setup.sh` / `setup.bat` (얼리지 않고 소스 실행) | 별도 1 |
| Windows 빌드 위치 | 맥 로컬 CI + **외부 Windows AI 머신(Syncthing 동기화)** | 2 파이프라인 |

산출물 실측(`dist/upload_installers/`):
- `aimax-bundle-macos.dmg` 43MB
- `aimax-bundle-windows.exe` 35MB
- `AIMAX-Office-Manager-Setup-0.1.5.exe` 161MB / `-portable.exe` 160MB
- `Pencil-Setup-1.0.0.exe` 88MB / `-portable.exe` 88MB
- `archive-windows-2026...` 폴더 다수 (v1039~v1044, 하루에 여러 번 재빌드한 흔적)

→ **빌드 한 번 = 위 축들의 곱.** 버그 하나가 어느 조합에서 터졌는지 추적 불가능에 가깝다.

## 2. 코드 레벨 충돌·중복 지점

### 2-1. 죽은 코드: Nuitka
- `build_nuitka.py` 헤더는 "CI 빌드: GitHub Actions(windows-latest)에서 동일 명령 실행"이라 적혀 있음.
- 그러나 실제 `.github/workflows/build.yml`은 **windows/macos 둘 다 `python build.py`(PyInstaller)만 실행** (build.yml:46, 104).
- 결론: **Nuitka는 문서상 살아있다고 적혀 있으나 실제 파이프라인에서 안 쓰임 = 죽은 코드.**
  유지하면 "어느 걸로 빌드했더라" 혼선만 유발.

### 2-2. 코드베이스 포크: split_version
- `split_version/app.py`(282KB)는 메인 `app.py`(293KB)를 **통째로 복제한 포크**.
- 두 복제본이 어긋나는 걸 막으려고 `scripts/preflight_split_drift.py`라는 **감시 도구를 따로 작성**해야 했음 (CI build.yml:44, 101에서 실행).
- 즉 한 기능 고치면 두 파일을 맞춰야 하고, 안 맞으면 preflight가 빌드를 막음.
- `split_version/`의 진입점 분화: `app_engage.py` / `app_engage_write.py` / `app_find.py` / `app_write.py`
  → 직원별·기능별 축소판을 각각 만들고 있음.
- 결론: **드리프트 위험을 감수하며 전체 앱을 2벌 유지 중. 유지보수 비용 2배.**

### 2-3. 브랜드 리브랜딩이 빌드 시점에 주입됨
- AIMAX-Office-Manager(지은), Pencil(나경) 등 이름은 `aimax_installer.iss`에
  `/DAppDisplayName` 파라미터로 외부에서 주입 (코드 내 정의 없음).
- Windows 쪽 Office-Manager/Pencil은 **외부 Windows AI가 별도로 빌드**해서 Syncthing으로 맥에 전달
  (`docs/windows_build_v1_0_1_handoff.md`, `docs/windows_ai_build_prompt.md`).
- 결론: **브랜드마다 별도 빌드. Windows는 사람(외부 AI)이 수동 개입하는 두 번째 파이프라인.**
  두 파이프라인이 어긋나면 "맥에선 되는데 윈도우 배포본은 구버전" 류 오류 발생.

### 2-4. PyInstaller hidden-import 두더지잡기
- `build.py:185~265`에 `--hidden-import` 40개 이상 + `--collect-submodules`/`--collect-all` 다수.
- 이 목록의 한 줄 한 줄이 **"PyInstaller가 자동으로 못 찾아서 사후에 때운 과거 런타임 크래시"**의 화석.
- 동적 import(예: `win32clipboard`, `PIL._tkinter_finder`, `keyring.backends.*`)는 누락 시
  **빌드는 성공하지만 고객 PC 런타임에서 터짐** → "내 맥에선 되는데" 전형.
- 결론: **얼린 번들 방식의 구조적 취약점.** 의존성 추가할 때마다 이 목록 관리 필요.

### 2-5. 두 갈래 실행 경로의 공존
- 경로 A(얼림): `build.py` → PyInstaller → (Win)Go런처+Inno / (Mac)dmg
- 경로 B(소스): `setup.sh`/`setup.bat` → 시스템 Python 탐색 → venv → 소스 실행
- 두 경로는 **환경 가정이 다름**:
  - A는 Python을 번들에 넣어 들고 감(대신 PyInstaller 취약점)
  - B는 사용자 PC의 Python에 의존(대신 hidden-import 문제 없음, 하지만 Python 못 찾으면 실패)
- 결론: **사용자에게 무엇을 주는지 자체가 일관되지 않음.** 둘을 동시에 유지하면 디버깅 표면 2배.

## 3. "환경 차이 오류"가 실제로 터지는 지점 (원인 지도)

| 증상 분류 | 근본 원인 | 위치 |
|---|---|---|
| 고객 PC에서만 크래시 | PyInstaller 동적 import 누락 | `build.py:185-265` |
| "Python을 찾을 수 없음" | 시스템 Python 탐색 실패(3.9 Tk 등) | `setup.sh:_find_python` |
| venv 깨짐(윈도우) | 한글/공백/OneDrive/롱패스 경로 | `setup.bat:[0/6]` |
| Gatekeeper 차단(맥) | ad-hoc 코드서명(`codesign -s -`) | `build.py:160-163` |
| "맥은 신버전, 윈도우는 구버전" | 두 빌드 파이프라인(로컬 CI vs 외부 AI) 드리프트 | `docs/windows_*build*` |
| split 빌드 실패/막힘 | 메인-split 코드 드리프트 | `scripts/preflight_split_drift.py` |

핵심: **환경 차이 오류의 대부분은 "얼린 번들(A경로)" + "시스템 Python 의존(B경로)" 두 군데서 나온다.**
Hermes가 이 둘을 모두 피하는 방식(소스 실행 + 런타임을 직접 들고 감 / 원격 스크립트)이 곧 처방의 방향.

## 4. 판정: 버릴 것 / 남길 것 / 통합할 것

### 버릴 것 (즉시, 위험 낮음)
1. **Nuitka (`build_nuitka.py`)** — CI에서 안 쓰임. 죽은 코드. 혼선만 유발.
2. **archive-windows-* 누적 폴더** — `dist/upload_installers/` 내 과거 빌드 잔해. 정리 대상.
3. (검토 후) **portable + Setup 이중 배포** 중 하나 — 둘 다 유지할 이유가 약하면 Setup 하나로.

### 남길 것 (핵심 자산 — 이미 Hermes 방식의 부품)
1. **`setup.sh` / `setup.bat`** — Hermes의 install.sh/install.ps1에 해당. B단계에서 uv로 강화할 토대.
2. **`aimax_agent_launcher.go`** — Hermes의 Tauri 런처에 해당. Windows 네이티브 런처로 유지 가치 있음.
3. **`aimax_installer.iss`** — Windows 설치 경험용. 유지.

### 통합할 것 (곱셈 → 덧셈으로)
1. **split_version 포크 → 단일 코드베이스 + 런타임 플래그.**
   현재: 전체 앱 2벌 복제 + 드리프트 감시 도구.
   목표: 메인 `app.py` 하나 + `--mode engage/write/find` 같은 진입 플래그로 분기.
   효과: preflight_split_drift.py 불필요, 유지보수 2배 → 1배.
2. **브랜드(AIMAX/예리/현주/지은/나경) → 빌드타임 설정값 1개로 외부화.**
   이미 `.iss`가 `/DAppDisplayName`으로 주입하는 구조 → 같은 패턴을 맥·소스 경로에도 통일.
   효과: 브랜드 N개여도 빌드 스크립트는 1개 + 설정 N줄.
3. **얼린 번들(A) vs 소스(B) → 하나로 결정.**
   권장: **B(소스 실행)를 주력으로, uv로 Python·의존성을 재현 가능하게.** (= Hermes 방식)
   A(PyInstaller)는 코드서명·공증이 꼭 필요한 배포 채널에만 한정 유지하거나 폐기.

## 5. 통합 후 목표 형태 (B단계 미리보기)

```
[사용자] 런처 실행 (맥: .command/.app / 윈도우: aimax_agent_launcher.exe)
   → setup 스크립트 호출
   → uv 가 고정 Python + 잠긴 의존성 설치 (시스템 Python·venv 깨짐 문제 제거)
   → 단일 app.py 를 브랜드/모드 플래그와 함께 실행
```

곱셈 폭발 → 단일 줄기:
- 빌드 도구 5 → (런처 빌드 1 + uv) 사실상 1.5
- 코드베이스 2 → 1
- 브랜드 5 → 설정 5줄(빌드 스크립트는 1)
- 얼림/소스 2 → 1(소스+uv)

## 6. 다음 단계 제안

- **B단계**: `setup.sh`/`setup.bat`를 uv 기반으로 전환 (환경 차이 오류 직접 감소). 가장 체감 큼.
- **C단계**: split_version 포크를 단일 코드베이스 + 모드 플래그로 통합 (유지보수 절반).
- **D단계**: Hermes식 "원격 스크립트 + 커밋 핀 고정"으로 수정→재배포 비용을 git push 한 번으로.

권장 순서: **B → C → D** (체감 효과 → 유지보수 → 배포 자동화 순).
단, C(포크 통합)는 동작 변경이 크므로 단계 게이트에서 CEO 확인 후 진행.
