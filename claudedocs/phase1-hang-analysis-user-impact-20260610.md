# Phase 1 — 행(hang) 원인 분석 + 실사용자 영향 평가 (2026-06-10)

## 1. 행 리포트(AIMAX 오류.txt) 원인 확정

대상: 이 Mac(운영 머신)의 v1.0.48, 2026-06-07, 2시간 11분 "무응답" — **고객 리포트 아님**.

스레드 분석 (스택샷 4개):
- 메인 스레드: `local_agent/runtime.py:240`의 headless 에이전트 메인 루프 `time.sleep(0.1)` — **설계된 폴링 루프로, 죽은 게 아님**. 다만 Tk mainloop 없이 Apple Event만 수동 펌프하므로 macOS가 "이벤트 미처리 앱"으로 판정해 행 리포트를 남김 (창 상호작용 시 비치볼 가능)
- 워커 스레드: `__select`에서 **7,870초(2.2h) 블로킹** — selenium→chromedriver HTTP(urllib3, 타임아웃 없음) 대기. 오늘 실기에서 재현한 "브라우저 사망 → driver.get 무한대기"와 동일 계열
- 나머지: 락 대기 1, NSEventThread 정상

결론: "앱이 죽는다"가 아니라 **"워커가 브라우저 사망 시 무한 대기 + 잡이 무음 정체"**가 본질. v1.0.51 좀비 보호(하트비트 기반)는 러너 사망만 커버하므로 이 케이스는 사각지대.
