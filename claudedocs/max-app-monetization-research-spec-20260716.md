# MAX 앱화·무료/유료 분리 리서치 스펙

작성: 2026-07-16 / 발주: 7/13 주간회의 최종 결정 ("자신 있게 팔 수 있는 것은 MAX뿐")
성격: 리서치·판단 자료 작성. 코드 변경 없음. 산출물은 CEO 결정용 비교표.

## 배경 (7/13 주간회의 결정 사항)

- MAX(일정·업무를 PC에서 단축키로 빠르게 관리하는 툴)는 팀이 실사용으로 검증한 유일한 판매 가능 자산으로 합의됨.
- 타겟 가설: "정리 못하는 직장인 / ADHD 성향 사용자".
- 장기 로드맵: 앱화(스토어 등록) + 무료/유료 기능 구분 + 해외 진출. 즉시 실행이 아니라 로드맵 확정용 조사가 이번 과제.
- 단기(이번 분기)는 현 기능 고도화·오가닉 홍보·노션 연동 유료화 우선 — 이 스펙과 별개 트랙.

## 과제 1. 스토어 등록 가능성 조사

현 배포 형태(파이썬 데스크톱 앱, PyInstaller/Nuitka 빌드 — `build.py`, `build_nuitka.py`)를 전제로, 채널별 요건·공수·제약을 표로 정리:

| 채널 | 조사 항목 |
|---|---|
| Mac App Store | 샌드박스 요건(현 기능 중 걸리는 것: 전역 단축키·접근성 권한·자동 실행 등), 공증(notarization)만으로 스토어 밖 배포하는 대안 |
| Microsoft Store | Win32 앱 등록 절차, 현 EXE 배포와의 차이 |
| Google Play | 데스크톱 앱이라 해당 없음 — 모바일 버전을 새로 만들 때만 유효. "플레이스토어"가 회의에서 언급됐으나 실제로는 모바일 리빌드 규모의 별건임을 명시할 것 |
| 스토어 밖 직접 배포 (현행 유지) | 코드사이닝 + 자동 업데이트 + 자체 결제 조합의 현실성 |

각 채널에 대해: 필수 요건 / 예상 공수 / 수수료 / 현 코드베이스에서 막히는 지점 3개 이내.

## 과제 2. 무료/유료 결제 분리 리서치

회의에서 "무료 기능과 유료 기능을 나눠 팔자"까지 합의됐고, 그 결제를 어떻게 붙일지가 이 과제다.

1. 기능 경계 초안: 현 MAX 기능 목록을 무료(습관성 핵심)/유료(노션 연동 등 파워 기능) 2단으로 나눈 초안 1개. 회의에서 노션 연동 유료화가 이미 후보로 언급됨.
2. 결제 방식 비교: 스토어 인앱결제(수수료 15~30%) vs 자체 결제(라이선스 키·Lemon Squeezy/Paddle/Gumroad류 MoR) vs 기존 카페24 연계. 각각 수수료·환불·구독 관리·한국 결제 수단 지원 기준.
3. 라이선스 기술 방식: 키 검증 서버 유무, 오프라인 grace period, 기존 엔타이틀먼트 체계(세무 직원 엔타이틀먼트 게이트)와의 재사용 가능성.

## 과제 3 (후순위). 해외 앱 바이럴·오가닉 사례 조사

회의에서 "해외에서 잘 팔린 앱의 바이럴 마케팅 벤치마킹"이 함께 잡혔다. MAX와 같은 급(1인·소팀 데스크톱 생산성 툴)이 오가닉으로 큰 사례 3~5개(예: 유사 카테고리의 단축키·일정 툴)를 골라, 무엇을 반복했는지(콘텐츠 포맷·채널·빈도)를 MAX 오가닉 콘텐츠 전략 초안으로 연결. 과제 1·2 산출물이 나온 뒤 착수.

## 산출물·검증 기준

- `claudedocs/`에 비교표 중심 리포트 1개 + CEO 결정 질문 3개 이내 (예: 스토어 진입 vs 직접 배포, 결제 방식, 무료/유료 경계).
- 모든 수수료·정책 수치는 공식 문서 기준으로 조사 시점 날짜를 병기 (스토어 정책은 자주 바뀜).
- 코드 변경·계정 생성·결제 연동 실작업은 이 스펙 범위 밖. CEO 결정 후 별도 스펙으로.
