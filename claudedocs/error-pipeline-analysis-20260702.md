## 2. 진단 요약

1. 실패의 절반 이상은 "고장"이 아니라 "사용자 조치 필요 상태에서의 무한 재시도". 사전 차단(preflight)이 없어서 실패로 기록되고, 안내가 닿지 않아 같은 실패가 반복된다.
2. 윈도우 오류율 2배의 실체는 (a) onedir 업데이트 부분 교체 (파일 잠금/백신 간섭 추정), (b) 실행기 워커 미시작/행. 둘 다 특정 가능하고 수리 가능한 결함.
3. 버전 파편화가 이미 고친 버그를 계속 되살린다 (v1.0.2~3 잔존).
4. app.py의 except-pass 41곳(침묵 실패)이 "그냥 안 된다"류 모호 보고의 공급원.

## 3. 제안 (우선순위순)

### P1. 서버만 수정 — 러너 배포 불필요, 즉시 효과
1. **Preflight 게이트**: 잡 생성/할당 전 검사 — 러너 버전>=MIN, 최근 네이버 로그인 성공 여부, AI 키 유효성(마지막 실패 코드 캐시), 결제 한도 상태. 통과 못하면 잡을 만들지 않고 조치 안내 화면으로.
2. **재시도 루프 차단**: 동일 사용자+동일 오류코드 3연속 실패 시 해당 작업 종류 자동 일시정지 + 조치 안내. (송이 구독 자동일시정지 패턴 재사용)
3. **Gemini transient 자동 재시도**: 지수 백오프 2~3회 후에만 실패 처리 (34건 소거 예상).
4. **미해결 보고 능동 알림**: waiting_user 상태를 이메일(기존 transactional 메일 경로)로 통지 + 앱/웹 접속 시 상단 배너. 조치 완료 감지 시 자동 종결(job_completed_after_report 확장).

### P2. 다음 러너 릴리스 — 윈도우 신뢰성
5. **스테이지드 업데이트**: 새 버전을 별도 폴더에 받고 검증(매니페스트 해시) 후 런처가 전환. 실행 중 폴더 덮어쓰기 금지 → import 불일치·액세스 거부 계열 소거.
6. **시작 시 무결성 자기검사**: 빌드 매니페스트(모듈별 해시/버전 스탬프) 대조, 불일치 시 실행 거부+재다운로드 안내. 부분 교체가 나도 즉시 명확한 오류로 표면화.
7. **워커 2차 좀비보호**: 잡 수신 후 N초 내 워커 스레드 미기동 시 실행기 자체 재시작. 하트비트에 진행 단계 포함, 서버는 단계 정지 감지 시 잡 실패+안내.
8. **except-pass 정리**: 침묵 실패 41곳에 최소한 오류코드+로그 남기기 (예리 사일런트실패 표면화 연장선).

### P3. 운영
9. **MIN_VERSION 상향 강제**: 스테이지드 업데이트 안착 후 구버전 잡 할당 차단(업데이트 안내만 허용). v1.0.2~3 잔존 제거.
10. **텔레그램 수리 에이전트 역할 재정의**: (a) 사용자 조치형 → 자동 안내로 종결, (b) 기존 버그 → KB 매칭+수정 버전 안내, (c) 신규 버그만 텔레그램 에스컬레이션 + 증거 번들 첨부. 민수에게 오는 알림을 신규/치명 건으로 축소.
11. **주간 오류 다이제스트**: evening report에 버전별/플랫폼별 실패율 추세 추가 — MIN 상향·릴리스 판단 근거.
