# AIMAX AI 직원 콘솔 전환 Phase Plan

작성일: 2026-05-05

중요: 2026-05-05 현재 이 UI 전환 phase는 “웹앱을 운영 콘솔처럼 보이게 만드는 작업”까지 완료되었지만, 제품 경험상 더 큰 문제가 남아 있다. 사용자가 설치 파일을 실행하면 기존 Tkinter 전체 프로그램이 열리므로 웹앱이 본체처럼 느껴지지 않는다. 다음 작업은 이 문서를 계속 밀기보다 `docs/local_agent_transition_handoff.md`를 기준으로 Local Agent 전용화 Phase L0/L1부터 진행한다.

## 목적

AIMAX 웹앱을 단순 작업 실행 화면에서 여러 AI 직원을 관리하는 운영 콘솔로 전환한다.

핵심 방향:

- AIMAX는 “자동화 프로그램”이 아니라 “AI 직원을 고용하고 관리하는 콘솔”처럼 보여야 한다.
- 예리/현주는 첫 직원이고, 이후 검수/리포트/예약/리서치/운영 보조 직원이 자연스럽게 추가될 수 있어야 한다.
- 실제 네이버 자동화 로직은 기존 로컬 실행기 구조를 유지한다.
- 웹앱은 직원 목록, 권한, 설정 상태, 작업 큐, 오류 보고, 업데이트를 한 화면 체계로 정리한다.

## Lazyweb MCP 참고 방향

Lazyweb MCP 검색에서 아래 패턴을 반영한다.

- Glean Agent Library: AI agent를 카드/카테고리로 탐색하는 구조
- Slack Marketplace: 검색, 카테고리, 활성/비활성 상태, 설치/권한 CTA
- Remote HR Core / Slack Atlas: 직원 프로필, 역할, 권한, 팀 디렉토리
- FloQast AI Agents: agent 관리 테이블, 상태, 생성 흐름
- Intercom: AI와 사람이 함께 일하는 support/workspace 감각
- Appcues/Userpilot: 시작 체크리스트, 설정 완료율, 온보딩 단계

적용 원칙:

- 특정 서비스의 화면을 복제하지 않는다.
- AIMAX는 마케팅 페이지가 아니라 업무용 SaaS 콘솔이다.
- 감성은 직원 프로필 사진과 역할 설명에서 주고, 조작 영역은 조용하고 명확하게 둔다.

## 목표 IA

최종 메뉴 구조:

1. 대시보드
2. 직원
3. 작업
4. 설정
5. 오류 보고
6. 업데이트

각 메뉴의 역할:

- 대시보드: 전체 연결 상태, 오늘 가능한 직원, 막힌 설정, 최근 작업, 다음 행동
- 직원: AI 직원 목록, 카테고리, 사용 가능 상태, 권한 상태, 직원 상세
- 작업: 작업 생성, 작업 목록, 실행 상태, 실패 사유, 재시도
- 설정: 로컬 실행기, 네이버 계정, AI Key, 직원별 필수 설정, 다운로드
- 오류 보고: 보고 폼, 접수 ID, 처리 상태, 추가 정보 필요 여부
- 업데이트: 웹앱/로컬 실행기 버전, 필수 업데이트, 설치파일 다운로드

## 핵심 데이터 모델

직원 정의는 웹앱 하드코딩에서 서버 응답 중심으로 옮긴다.

예시:

```json
{
  "code": "yeri",
  "name": "예리",
  "role": "소통글쓰기 직원",
  "category": "content",
  "product": "yeri",
  "job_kind": "yeri_write",
  "execution": "local_agent",
  "status": "available",
  "required_settings": ["naver_account", "ai_key"],
  "profile_image": "/assets/avatar_yeri.jpg",
  "short_description": "키워드와 CTA를 바탕으로 네이버 블로그 글을 작성합니다."
}
```

상태값:

- `available`: 사용 가능
- `needs_setup`: 설정 필요
- `no_entitlement`: 권한 없음
- `agent_offline`: 로컬 실행기 미연결
- `unsupported_agent`: 현재 설치 실행기에서 미제공
- `update_required`: 업데이트 필요

## Phase 0. 제품 구조 확정

목적: 화면을 바꾸기 전에 AIMAX의 제품 은유와 정보 구조를 고정한다.

범위:

- “AI 직원 사무실 + 운영 콘솔” 방향 확정
- 메뉴 구조 확정
- 직원 카드/상세 정보 항목 확정
- 기존 대시보드에서 어떤 요소를 직원/작업/설정으로 옮길지 결정

산출물:

- 이 문서 확정본
- 화면별 wireframe 수준 구조

검증:

- 예리/현주/통합 구매자가 각각 어디서 무엇을 해야 하는지 설명 가능
- 새 직원 1명을 추가했을 때 어느 화면에 붙는지 설명 가능

## Phase 1. 서버 직원 카탈로그 정리

목적: 웹앱이 직원 정보를 하드코딩하지 않고 서버의 직원 카탈로그를 기준으로 그리게 한다.

범위:

- `/api/workers` 응답 확장
- 직원 코드, 이름, 역할, 상품 권한, 작업 타입, 실행 위치, 필수 설정, 이미지 경로 포함
- 기존 `yeri_writer`, `hyunju_sales`와 웹앱 `yeri`, `hyunju` 명칭 정리
- 권한/상태 계산에 필요한 서버 필드 정의

산출물:

- 확장된 `/api/workers`
- 서버 worker catalog 스키마

검증:

- 예리/현주가 서버 응답으로 내려옴
- 권한 없는 사용자에게 직원은 보이되 “구매 필요/권한 없음”으로 표시 가능
- 기존 job 생성 API는 깨지지 않음

진행 기록:

- 2026-05-05: `/api/workers` 응답에 `catalog_version`, `staff_code`, `product`, `job_kind`, `execution`, `required_settings`, 프로필 이미지, 설명, capability를 추가
- 2026-05-05: `/api/workers` 응답에 `job_kinds` 카탈로그 추가
- 2026-05-05: 공개 job 응답에 `staff_code`, `worker_name` 추가
- 2026-05-05: 웹앱이 `/api/workers`를 읽어 기존 예리/현주 카드 메타데이터에 반영하는 어댑터 초안 추가
- 2026-05-05: 운영 배포 완료, `https://api.aimax.ai.kr/api/workers`에서 예리/현주 카탈로그 응답 확인
- Windows/macOS 로컬 실행기 변경 없음: readiness key와 job kind는 기존 `yeri_write`, `hyunju_find`를 유지

## Phase 2. 웹앱 데이터 어댑터

목적: 직원/권한/readiness/job_kind를 한 곳에서 계산하게 해 UI 변경의 기반을 만든다.

범위:

- 웹앱에서 `employees` 하드코딩 최소화
- 서버 `/api/workers`, `/api/auth/me`, `/api/agent/status`를 합쳐 직원 상태 계산
- `canUseEmployee`, `jobBlockReason`, `employeeAction`을 일반화
- 예리/현주 전용 분기 축소

산출물:

- 직원 상태 계산 어댑터
- 직원별 CTA 계산 함수

검증:

- 예리 단독, 현주 단독, 통합 계정에서 직원 상태가 각각 맞게 표시
- 로컬 실행기 미연결, 설정 필요, 권한 없음 상태가 구분됨

진행 기록:

- 2026-05-05: 웹앱 내부에 직원 카탈로그 기반 helper 추가
  - `employeeEntries`
  - `employeeForJobKind`
  - `hasEmployeeEntitlement`
  - `settingState`
  - `entitledEmployees`
- 2026-05-05: `setupItems`가 예리/현주 전용 분기 대신 직원별 `requiredSettings`를 합쳐 설정 체크리스트를 구성하도록 변경
- 2026-05-05: `nextAction`이 예리/현주 하드코딩 대신 사용 가능한 직원과 첫 번째 차단 사유를 기준으로 안내하도록 변경
- 2026-05-05: 직원 카드/상세 화면이 카탈로그의 이름, 역할, 프로필 이미지, capability를 반영하도록 변경
- 2026-05-05: 작업 폼 활성화 로직을 `jobFormConfigs` 기반으로 정리
- 2026-05-05: 운영 `/app` 배포 완료, 공개 스크립트 파싱 통과
- Windows/macOS 로컬 실행기 변경 없음: job kind, readiness key, command contract는 유지

## Phase 3. 직원 콘솔 UI

목적: 앞으로 직원이 늘어나도 자연스럽게 확장되는 핵심 화면을 만든다.

범위:

- `직원` 탭을 별도 주요 메뉴로 구성
- 직원 카드 grid/list
- 검색 또는 카테고리 필터 초안
- 직원 상세 패널
- 프로필 사진, 역할, 필요한 설정, 마지막 작업, 대표 CTA 표시

산출물:

- 직원 목록 화면
- 직원 상세 화면

검증:

- 모바일/데스크톱에서 직원 카드가 깨지지 않음
- 직원이 2명에서 8명으로 늘어나도 레이아웃이 유지됨
- 프로필 사진이 실제 직원 구분에 도움이 됨

진행 기록:

- 2026-05-05: 웹앱 좌측 내비게이션에 `직원` 탭 추가
- 2026-05-05: 직원 탭에 전체/사용 가능/설정 필요/권한 없음 요약 카드 추가
- 2026-05-05: 직원 검색 입력과 상태 필터(`전체`, `사용 가능`, `설정 필요`, `권한 없음`) 추가
- 2026-05-05: 카탈로그 기반 직원 카드 grid 추가
- 2026-05-05: 직원 상세 패널 추가
  - 프로필 사진
  - 역할/상태
  - 필요 설정
  - 대표 업무
  - 최근 작업
  - 대표 CTA
- 2026-05-05: 운영 `/app` 배포 완료, 공개 스크립트 파싱 통과
- Windows/macOS 로컬 실행기 변경 없음: 웹앱 UI만 변경

## Phase 3.5. Visual Refresh

목적: 구조만 바뀐 웹앱을 실제 제품처럼 보이게 시각 스타일을 개선한다.

범위:

- 전체 색감, 배경, 패널, 버튼, 상태 pill, 카드 hierarchy 개선
- 대시보드 첫인상을 “운영 현황판”처럼 정리
- 직원 탭을 “AI 직원 사무실 + 직원 디렉토리” 느낌으로 강화
- 프로필 사진과 직원 상태가 더 잘 보이도록 카드 시각 강조
- 기존 기능 계약, job kind, readiness key, 서버 API는 변경하지 않음

산출물:

- `oracle/aimax-reports-api/static/app.html` visual refresh

검증:

- 기존 로그인/다운로드/설정/작업/오류 보고 UI 요소가 유지됨
- 모바일에서 nav, 직원 카드, 상세 패널 텍스트가 겹치지 않음
- 운영 `/app` 스크립트 파싱 통과

진행 기록:

- 2026-05-05: Visual Refresh 운영 배포 완료
- 전체 팔레트를 기존 navy/blue 중심에서 차콜/그린/뉴트럴/보조 블루 accent로 재정의
- 사이드바, 브랜드 마크, 활성 메뉴 상태 개선
- 패널/카드에 subtle shadow와 더 또렷한 border hierarchy 적용
- 대시보드 `next-action`을 운영 현황판의 primary action처럼 강조
- 직원 카드에 상단 accent bar, hover elevation, active state 개선
- 직원 상세 패널에 shadow와 white panel 톤 적용
- 모바일 nav를 5열에서 3열 wrap 구조로 조정해 텍스트 압박 완화
- 운영 `/app`에서 Visual Refresh CSS 서빙 확인
- 운영 `/app` 공개 스크립트 파싱 통과
- Windows/macOS 로컬 실행기 변경 없음: 웹앱 CSS/UI만 변경

## Phase 4. 작업 화면 재구성

목적: 작업 생성과 작업 상태를 직원 중심으로 정리한다.

범위:

- `작업` 탭에 작업 생성 + 작업 목록 통합
- 직원 선택 후 해당 직원 작업 폼 표시
- 준비 안 된 직원은 폼 대신 차단 사유와 해결 CTA 표시
- 최근 작업/실패 작업/진행 작업 구분

산출물:

- 직원 기반 작업 생성 화면
- 작업 목록 상태 UI

검증:

- 기존 예리/현주 작업 생성 API와 로컬 Agent dispatch가 유지됨
- 설정 부족 상태에서 작업 생성이 차단됨
- 실패 작업의 마지막 로그가 보임

진행 기록:

- 2026-05-05: 대시보드 하단에 있던 `작업 지시` 폼을 `작업` 탭으로 이동
- 2026-05-05: `작업` 탭 상단에 `작업 생성` 패널 추가
- 2026-05-05: 직원 선택 카드(`jobEmployeeSwitch`) 추가
- 2026-05-05: 선택한 직원의 작업 폼만 표시되도록 `selectedJobKind` 기반 렌더링 추가
- 2026-05-05: 직원 카드/상세 패널의 `작업 지시하기` CTA가 해당 직원 작업 폼으로 이동하도록 연결
- 2026-05-05: 작업 목록은 `작업 생성` 아래에 유지
- 2026-05-05: 운영 `/app` 배포 완료, 공개 스크립트 파싱 통과
- Windows/macOS 로컬 실행기 변경 없음: 기존 `POST /api/jobs`, job kind, readiness key 유지

## Phase 5. 대시보드 재정의

목적: 대시보드를 잡다한 기능 모음이 아니라 오늘의 운영 상태 화면으로 정리한다.

범위:

- 다음 행동
- 전체 직원 상태 요약
- 로컬 실행기 연결
- 오늘/최근 작업
- 막힌 설정
- 오류 보고 바로가기

산출물:

- 운영 현황형 대시보드

검증:

- 사용자가 첫 화면에서 “지금 무엇을 해야 하는지” 바로 이해함
- 작업 폼은 대시보드에서 과도하게 길게 노출되지 않음

진행 기록:

- 2026-05-05: 대시보드 상단에 `운영 현황` 요약 카드 추가
  - 사용 가능한 직원
  - 설정이 필요한 직원
  - 진행 중인 작업
  - 확인이 필요한 작업
- 2026-05-05: `막힌 항목` 패널 추가
  - 비밀번호 변경, 로컬 실행기 연결, 네이버 계정, AI Key, 서로이웃 멘트 등 readiness 기반 차단 항목 표시
  - 직원별 차단 사유도 함께 요약
- 2026-05-05: `빠른 이동` 패널 추가
  - 직원
  - 작업
  - 설정
  - 오류 보고
- 2026-05-05: 작업 생성 폼은 대시보드에 다시 넣지 않고 `작업` 탭에 유지
- Windows/macOS 로컬 실행기 변경 없음: 웹앱 대시보드 렌더링만 변경

## Phase 6. 설정/업데이트 분리

## Phase 7. 오류 보고 운영화

- 오류 상태: `new`, `checking`, `fixed`, `need_user_info`
- 최근 내가 보낸 오류 목록
- 추가 정보 필요 시 안내

산출물:

- 오류 보고 목록/상태 UI
- 관리자 콘솔 Phase C와 연결 가능한 데이터 구조

검증:

## Phase 8. 운영 교체

범위:

- 새 UI를 `/app-next` 또는 로컬에서 먼저 smoke

- 통과 후 `/app` 교체

산출물:

- 운영 반영된 AI 직원 콘솔

검증:

- macOS/Windows 다운로드 옵션 유지
- 기존 로컬 Agent job dispatch 유지
- 비밀번호 변경/오류 보고 회귀 없음

## MVP 구현 범위 제안

첫 구현은 Phase 1~5까지만 묶는다.

- 서버 직원 카탈로그 확장
- 웹앱 직원 상태 계산 일반화
- 직원 탭 신설
- 작업 탭 정리
- 대시보드 재정의

Phase 6~7은 다음 배포에서 이어간다. 이렇게 하면 현재 구매자가 쓰는 기능은 유지하면서도, 제품 첫인상이 “AI 직원 콘솔”로 바뀐다.
