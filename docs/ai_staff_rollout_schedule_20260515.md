# AIMAX 신규 직원 사용 가능 일정안

작성일: 2026-05-15  
기준: 2026-05-15부터 바로 착수, 개발자 1명 중심, 배포/QA 병행 가능하다는 전제

## 1. 대표님께 전달할 한 줄 요약

신규 직원 5명은 한 번에 완성 배포하기보다, 2026-05-20에 관리자 기반을 먼저 배포하고 2026-05-29부터 윤미를 시작으로 매주 1명씩 순차 베타 배포하는 일정이 가장 현실적입니다. 전체 직원이 최소 사용 가능한 상태가 되는 목표일은 2026-07-03입니다.

## 2. 사용 가능 단계 정의

| 단계 | 의미 | 사용자 범위 |
| --- | --- | --- |
| 카드 노출 | 직원 프로필과 준비 상태만 보임 | 전체 또는 지정 사용자 |
| 내부 알파 | 관리자/팀이 실제 기능을 테스트 가능 | 내부팀 |
| 제한 베타 | 일부 고객 또는 대표 지정 계정에서 사용 가능 | 베타 사용자 |
| 일반 배포 후보 | 핵심 기능, 로그, 오류 대응, 권한 제어가 갖춰짐 | 전체 또는 상품 권한 사용자 |

현재 상태:
- 신규 5명은 카드 노출 단계다.
- 기능 실행은 아직 막혀 있다.
- 다음 단계는 관리자에서 상태/노출/권한을 제어할 수 있는 기반을 만드는 것이다.

## 3. 전체 일정표

| 기간 | 목표 | 배포 가능 결과 |
| --- | --- | --- |
| 2026-05-15 ~ 2026-05-20 | 직원 카탈로그/관리자 기반 | 관리자에서 직원 상태, 노출, 정렬, 설명 관리 |
| 2026-05-21 ~ 2026-05-29 | 윤미 1차 | 스크립트작가 윤미 제한 베타 |
| 2026-05-27 ~ 2026-06-05 | 송이 1차 | 자료조사원 송이 제한 베타 |
| 2026-06-03 ~ 2026-06-12 | 현성 1차 | PM 현성 제한 베타 |
| 2026-06-10 ~ 2026-06-24 | 상수 1차 | 회계원 상수 관리자/제한 베타 |
| 2026-06-17 ~ 2026-07-03 | 나경 1차 | 판서쌤 나경 로컬 앱 파일럿 |

## 4. 단계별 상세 일정

### 0단계. 직원 운영 기반

기간:
- 2026-05-15 ~ 2026-05-20

목표:
- 신규 직원들을 코드 수정 없이 관리자 페이지에서 관리한다.
- 어떤 직원이 준비 중인지, 베타인지, 사용 가능한지 상태를 바꿀 수 있게 한다.

산출물:
- `staff-catalog.json` 기반 직원 카탈로그
- 관리자 직원 관리 화면
- `/api/workers` 호환 유지
- 실행 가능 직원과 준비 중 직원 분리

배포 판단:
- 2026-05-20 운영 배포 가능

대표님께 보여줄 수 있는 것:
- 관리자 페이지에서 직원 상태를 직접 바꾸는 화면
- 사용자 화면에서 준비 중 직원과 사용 가능 직원이 구분되는 모습

### 1단계. 윤미: 스크립트작가

기간:
- 내부 알파: 2026-05-22
- 제한 베타: 2026-05-29
- 일반 배포 후보: 2026-06-03

첫 버전 기능:
- 레퍼런스 URL 또는 텍스트 입력
- 핵심 패턴 분석
- A/B/C 스크립트 초안 생성
- 결과 저장/복사

준비해야 할 것:
- AI 모델/비용 정책 확정
- 유료 호출 전 확인 UI
- 자동 재시도 금지
- 요청 ID와 오류 로그 저장
- 관리자에서 윤미 활성화/비활성화 가능

왜 1순위인가:
- 레포에 이미 AI 생성 로직과 결과 스키마가 있다.
- 고객이 바로 체감하기 쉽다.
- 송이와 연결했을 때 콘텐츠 제작 흐름이 자연스럽다.

### 2단계. 송이: 자료조사원

기간:
- 내부 알파: 2026-05-29
- 제한 베타: 2026-06-05
- 일반 배포 후보: 2026-06-10

첫 버전 기능:
- URL/텍스트/메모 기반 레퍼런스 등록
- 플랫폼, 카테고리, 태그 정리
- 간단 분석 요약
- 윤미에게 넘길 수 있는 자료 묶음 생성

첫 버전에서 제외:
- 자동 크롤링
- 플랫폼별 로그인 수집
- 대량 수집 자동화

준비해야 할 것:
- `research_items` 또는 `references` 저장 구조
- 윤미와 공유할 레퍼런스 데이터 포맷
- 저작권/플랫폼 리스크를 피하는 수동 입력 중심 UX

### 3단계. 현성: PM

기간:
- 내부 알파: 2026-06-05
- 제한 베타: 2026-06-12
- 일반 배포 후보: 2026-06-17

첫 버전 기능:
- 프로젝트/작업/담당 직원 관리
- 마감일, 상태, 변경 사유 기록
- 직원별 대기 작업/실패 작업 요약
- 관리자용 운영 보드

첫 버전 방향:
- 고객용 프로젝트 관리툴보다 내부 운영 PM에 가깝게 시작한다.
- AIMAX 직원들이 많아질수록 운영 상태를 한눈에 보기 위한 역할이다.

준비해야 할 것:
- 기존 localStorage 구조를 서버 저장 구조로 전환
- 직원 카탈로그, 작업 로그와 연결
- 관리자 권한 중심 화면

### 4단계. 상수: 회계원

기간:
- 내부 알파: 2026-06-12
- 제한 베타: 2026-06-24
- 일반 배포 후보: 2026-07-01

첫 버전 기능:
- 매출/비용 수동 입력
- 상품/채널별 요약
- 간단 손익 대시보드
- CSV 내보내기

주의:
- 회계 데이터는 민감하다.
- 상수는 기능보다 권한, 백업, 감사 로그가 먼저다.
- 일반 사용자 배포는 다른 직원보다 보수적으로 잡아야 한다.

준비해야 할 것:
- 데이터 수정/삭제 이력
- 관리자/사용자 권한 구분
- CSV 백업/내보내기
- 샘플 데이터 모드

### 5단계. 나경: 판서쌤

기간:
- 내부 알파: 2026-06-19
- Mac 파일럿: 2026-06-26
- 일반 배포 후보: 2026-07-03

첫 버전 기능:
- AIMAX에서 나경 직원 카드와 설치 상태 표시
- Mac 앱 다운로드 링크
- 앱 버전 표시
- 수동 설치/실행 안내

첫 버전에서 제외:
- 완전 자동 업데이트
- 실시간 웹 연동
- 계정 페어링
- 녹화/판서 결과 자동 업로드

주의:
- 나경은 웹 기능이 아니라 Electron 데스크톱 앱이다.
- 코드 병합보다 빌드, 서명, 다운로드, 버전 관리가 핵심이다.

## 5. 우선순위

1. 관리자 기반
2. 윤미
3. 송이
4. 현성
5. 상수
6. 나경

이유:
- 윤미와 송이는 고객이 가장 빨리 가치를 느낄 수 있다.
- 현성은 내부 운영 효율을 높인다.
- 상수는 가치가 크지만 재무 데이터라 보수적 배포가 필요하다.
- 나경은 웹 통합이 아니라 별도 앱 배포라 준비 항목이 다르다.

## 6. 대표님께 말할 때의 권장 표현

2026-05-20까지는 직원들을 실제 운영할 수 있는 관리자 기반을 먼저 배포하겠습니다. 그 다음 2026-05-29 윤미를 시작으로, 송이, 현성, 상수, 나경 순서로 매주 베타 배포를 열겠습니다. 완성형은 아니더라도 2026-06-05에는 콘텐츠 쪽 직원 2명, 2026-06-12에는 운영 PM까지, 2026-07-03에는 신규 5명 전체를 최소 사용 가능한 상태로 만드는 일정이 현실적입니다.

## 7. 일정 리스크

윤미:
- AI 모델 비용과 품질 조정이 필요하다.
- 유료 호출 정책이 늦어지면 베타가 2~3일 밀릴 수 있다.

송이:
- 자동 수집을 첫 버전에 넣으면 일정이 크게 늘어난다.
- 첫 버전은 수동 레퍼런스 정리로 제한해야 한다.

현성:
- 내부 운영용인지 고객용 PM인지 범위가 흔들리면 일정이 밀린다.
- 첫 버전은 내부 운영용으로 고정해야 한다.

상수:
- 회계 데이터 권한/백업/감사 로그가 없으면 실사용 배포하면 안 된다.
- 제한 베타까지만 먼저 열고 일반 배포는 신중히 판단한다.

나경:
- Electron 앱은 웹 배포보다 변수가 많다.
- 앱 서명, 다운로드, OS 권한, 설치 안내가 일정 리스크다.
