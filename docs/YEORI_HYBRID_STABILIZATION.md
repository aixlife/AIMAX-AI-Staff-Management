# AIMAX R3 예리 (블로그 글쓰기) Hybrid 안정화 설계 보고서

> **문서 목적**: macOS와 Windows 11 양쪽 환경에서 작동하는 예리(블로그 글쓰기 에이전트) 로컬 실행기가 네이버 스마트에디터 One에 글과 이미지를 오차 없이 안정적으로 입력할 수 있도록 OS/브라우저 자동화 관점에서 리스크를 분석하고 자가 복구(Self-healing) 및 미니멀 UI UX 대응 시나리오를 설계합니다.
> **작성 기준**: 유료 API 호출 배제, 실제 네이버 발행/저장 테스트 배제, 개인정보/보안 토큰 유출 원천 금지 규칙을 엄격히 준수합니다.

## 1. OS 자동화 리스크 분석 (Automation Risks)

네이버 스마트에디터 One은 일반적인 HTML 구조와 달리 고도의 보안 및 복잡한 컴포넌트 구조를 채택하고 있어 다음 3대 자동화 리스크를 가집니다.

```
┌──────────────────────────────────────────────────────────┐
│             네이버 스마트에디터 One 자동화 장벽              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  [장벽 1] Iframe & Shadow DOM  ➔  직접적인 DOM 접근 제한     │
│  [장벽 2] 파일 다이얼로그 차단 ➔  OS 시스템 팝업 제어 불가   │
│  [장벽 3] 봇 탐지 솔루션 가동   ➔  지연시간 고정 시 IP 차단  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

1.  **Iframe 및 내부 동적 렌더링**
    - 스마트에디터 One은 글쓰기 영역 전체가 단일 혹은 다중 **Iframe** 내부에 격리되어 있으며, 실시간 React/Redux 상태에 따라 내부 요소가 동적으로 생성/소멸합니다. 
    - 단순 엘리먼트 쿼리(`page.locator()`)로는 Iframe 장벽에 막혀 `TimeoutError`를 유발합니다.
2.  **OS 파일 선택 다이얼로그 제어 불가능**
    - 이미지 삽입 버튼을 클릭하면 OS 네이티브 파일 선택 창(Mac Finder / Windows File Explorer)이 뜹니다. 
    - Playwright/Selenium은 웹 브라우저 외부에 있는 OS 시스템 팝업 창을 직접 클릭하거나 조작할 수 없어 자동화 흐름이 무한 프리징됩니다.
3.  **지능형 봇 탐지 솔루션 (Anti-Bot Compliance)**
    - 네이버는 대량 스팸 방지를 위해 동작 속도, 마우스 궤적, 키보드 입력 간격 등을 실시간 모니터링합니다. 
    - 일정한 주기로 본문을 타이핑하거나 즉시 입력 주입 시, 비정상 봇으로 감지되어 임시 차단 및 보호 조치가 실행됩니다.

## 2. macOS vs Windows 11 기술적 차이점 극복 설계

양대 운영체제에서 공통 코드로 실행하기 위해 반드시 청산해야 하는 3대 물리적 격차 극복 설계입니다.

### ① 키보드 레이아웃 & IME(한영 입력) 및 복사-붙여넣기 격차
- **문제**: 본문 복사-붙여넣기 시 Mac은 `Command + V`, Windows는 `Control + V` 단축키를 씁니다. 또한 키보드 타이핑(`page.type()`) 모듈 작동 시 로컬 PC의 현재 한영 상태가 영어 모드인지 한글 모드인지에 따라 `안녕하세요`가 `dkssudgktpdy`로 오타 매핑되거나 텍스트가 심각하게 깨집니다.
- **해결**: 단축키와 입력 이벤트를 OS 단독 분기 클래스 `OSKeyboardBridge`로 통일합니다.
  - **단축키 분기**: `OSKeyboardBridge.get_paste_shortcut()`을 제공하여 OS에 맞게 `Meta+V`(Mac)와 `Control+V`(Win)를 동적으로 매핑합니다.
  - **IME 극복**: 느리고 오타 위험이 큰 단순 키보드 타이핑 방식을 원천 배제합니다. 대신 본문 노드의 DOM에 클립보드 API 혹은 Javascript 실행문을 통해 텍스트 데이터 덩어리를 직접 복사 주입(`document.execCommand('insertText', false, text)`)함으로써, 로컬 PC의 한/영 키 상태에 상관없이 원문 그대로 고속 주입합니다.

### ② 디스플레이 해상도 / DPI 및 창 크기 편차
- **문제**: Mac의 Retina 디스플레이는 고배율 픽셀 매핑(`devicePixelRatio: 2.0`)을 쓰고, Windows는 스케일링 비율(1.0, 1.25, 1.5)이 제각각입니다. 이 때문에 물리적인 클릭 좌표(X, Y)를 계산하여 클릭하면 엉뚱한 허공을 눌러 동작 실패를 유발합니다.
- **해결**: 
  - 물리 좌표(Absolute Coordinates) 클릭을 전면 배제합니다.
  - 브라우저 실행 시 창 크기를 OS 종류에 관계없이 동일한 가상 뷰포트(예: `viewport={width: 1440, height: 900}`)로 고정합니다.
  - 요소 클릭 시 좌표가 아닌 엘리먼트 바운딩 박스 기준의 상대 좌표(`locator.click()`)를 호출하고, 폴백 클릭 시에는 Javascript의 `.click()` 이벤트를 DOM에 직접 쏘는 방식으로 DPI 종속성을 완전히 무력화합니다.

### ③ OS별 파일 경로 체계 표준화
- **문제**: 이미지 파일 경로가 Mac은 POSIX 슬래시(`/Users/...`) 체계이고 Windows는 역슬래시(`C:\Users\...`) 체계이므로 경로 해석 에러가 발생합니다.
- **해결**: 모든 로컬 파일 경로는 Python 내장 `pathlib.Path` 라이브러리를 사용하여 로컬 구동 OS에 맞춰 자동 형변환(POSIX ➡️ UNC) 처리되도록 규격화합니다.

## 3. 자가 복구(Self-healing) 아이디어

에러율을 0%에 가깝게 만들기 위한 예리 로컬 실행기의 4가지 자가 복구 시나리오입니다.

```
                  ┌─────────────────────────────────────────┐
                  │      Smart Editor Self-healing Loop     │
                  └────────────────────┬────────────────────┘
                                       │
         ┌─────────────────────────────┼─────────────────────────────┐
         ▼                             ▼                             ▼
┌──────────────────┐         ┌──────────────────┐          ┌──────────────────┐
│  1단계: 셀렉터   │         │  2단계: DOM      │          │  3단계: 이미지   │
│  동적 폴백 배열  │         │  구조 횡단 파싱  │          │  드래그앤드롭   │
└──────────────────┘         └──────────────────┘          └──────────────────┘
```

### ① Smart Editor 셀렉터 실패 대비: 3중 폴백 로케이터
네이버 스마트에디터의 ID나 Class명이 동적으로 바뀌어 타임아웃이 발생할 경우, 다음 3중 검색 필터를 순차적으로 가동해 스스로 요소를 찾아냅니다.
1.  **CSS/XPath Fallback 배열**: 미리 선언해 둔 3~4개의 대안 XPath 배열을 순차 호출합니다.
    - 예: `[ "div.se-edit-area", "div[contenteditable='true']", "iframe#se3_iframe >>> div.se-document-body" ]`
2.  **텍스트 및 속성 기반 탐색**: 클래스명이 바뀌어도 변하지 않는 텍스트 알맹이(예: *"본문에 내용을 입력하세요"*)나 ARIA 역할 속성(`role="textbox"`, `placeholder="..."`)을 횡단 검색하여 타겟을 역추적합니다.
3.  **Javascript DOM 탐색**: Playwright API가 요소를 놓치더라도, 브라우저 내부 Javascript 컨텍스트에서 `document.querySelector` 계층을 직접 순회하여 요소를 반환받아 강제 클릭을 유입시킵니다.

### ② 이미지 삽입 안정성 확보: 파일 인풋 가로채기 & 드래그앤드롭 시뮬레이터
OS 네이티브 파일 팝업 창을 우회하여 백그라운드에서 이미지를 무중단 주입합니다.
1.  **File Input Event Interception**: 
    - 스마트에디터의 이미지 삽입 버튼 뒤에 숨겨진 실제 `<input type="file">` 태그를 찾아내어 Playwright의 `page.setInputFiles(selector, file_paths)`를 통해 직접 파일 스트림을 꽂아 넣습니다. OS 파일 다이얼로그 팝업이 뜨는 것을 원천 원격 차단합니다.
2.  **Drag and Drop JS Simulator (폴백)**:
    - 만약 인풋 태그가 막혔을 경우, 브라우저 내부 DOM에 가상의 `DataTransfer` 객체를 생성하고, 파일 데이터 스트림을 담아 본문 영역 위로 **DragEnter ➡️ DragOver ➡️ Drop** 자바스크립트 이벤트를 트리거하여 브라우저 수준에서 드롭다운을 무인 완성시킵니다.

### ③ 유료 API 없는 로컬 좌표 자가 진단
- 유료 API 사용 없이, 엘리먼트 획득 실패 시 현재 브라우저의 전체 DOM 트리 구조를 JSON으로 직렬화(Serialize)하여 **로컬 자가 진단 모듈(Self-Diagnoser)**에 넣습니다. 
- 진단 모듈은 엘리먼트 중복 배치나 iframe 포커스 빗나감 상태를 로컬 룰 베이스(Rule-base) 알고리즘으로 분석하여, 프레임을 자동으로 변경(`frame.child_frame()`)하거나 스크롤을 엘리먼트 위치로 강제 이동(`scrollIntoViewIfNeeded()`)한 후 재시도합니다.

## 4. 모니터링 및 사용자 공감형 UX 설계 (Status & Logs)

자동화가 실행되는 동안 사용자가 **"지금 살아있는지/멈췄는지/대기해야 하는지"**를 단 1초의 무반응 시간 없이 명확하게 인지할 수 있는 투명한 UX 장치를 마련합니다.

```
┌──────────────────────────────────────────────────────────────────────────┐
│ 예리 블로그 발행 현황                                                     │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ● [3단계] 본문 작성 및 이미지 배치 진행 중...                              │
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░ [60%]                                │
│                                                                          │
│  ※ 안전 지연 시간 준수 중 (4초 후 다음 단계로 이행합니다.)                  │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### ① 실시간 5단계 한글 상태바 UX (WebSocket 스트리밍)
로딩 서클만 무한히 도는 무반응 상태를 전면 배제하고, 로컬 에이전트의 현재 공정률을 웹소켓(IPC)으로 어드민 UI에 실시간 스트리밍합니다.
- **1단계**: `● [준비] 로컬 네이버 로그인 세션 확인 및 브라우저 기동 중...`
- **2단계**: `● [진입] 블로그 글쓰기 에디터 접속 완료. 본문 프레임 탐색 중...`
- **3단계**: `● [작성] 고품질 텍스트 복사-붙여넣기 및 이미지(N장) 파일 바인딩 중... (N%)`
- **4단계**: `● [조율] 태그 등록, 카테고리 매칭 및 스마트 에디터 정렬 세팅 중...`
- **5단계**: `● [완료] 포스팅 준비 완료! 대표님의 최종 검토 및 승인을 기다립니다.`

### ② 무정지 디버그 스냅샷 (Diagnostics Bundle)
동작이 실패하거나 사용자 개입이 필요한 임계점에 도달한 순간, 로컬 에이전트는 `/diagnostics/` 디렉토리에 **무정지 디버그 번들**을 자동 덤프합니다.
- **screenshot_timestamp.png**: 에러 시점의 전체 브라우저 렌더링 스크린샷 (DPI 배율 보정 저장).
- **dom_tree_timestamp.html**: 에러 시점의 iframe 내부를 포함한 전체 DOM 원본 덤프.
- **agent_console_timestamp.log**: 크롬 개발자 도구 콘솔 로그 및 네트워크 타임라인 로그.
- **error_context.json**: 실패한 단계명, 실패 타겟 셀렉터, 기기 정보(OS, 창 크기, 라이선스 상태).

## 5. 구현 우선순위 로드맵 (Priority Roadmap)

이 안정화 구조를 안전하게 가동하기 위한 로직 구현의 우선순위 단계입니다.

| 우선순위 | 작업 항목 (Task) | 해당 중점 사항 | 핵심 목표 |
| :---: | :--- | :--- | :--- |
| **P0** | **IME 무력화 복사-붙여넣기 코어** | IME/한영 입력 극복 | 로컬 한영키 상태와 무관한 고속/고품질 본문 타이핑 완성 |
| **P0** | **File Input 강제 바인딩** | 이미지 삽입 안정성 | OS 파일 다이얼로그 팝업 원천 차단 및 무인 이미지 삽입 |
| **P1** | **Iframe Dynamic Selector Fallback**| 셀렉터 실패 대비 | 네이버 UI 미세 변경 시 크래시 없이 자동 요소를 역추적 |
| **P1** | **실시간 5단계 상태 스트리머** | 사용자 인지 UX | 사용자가 언제든 어느 단계에 머물러 있는지 실시간 한글 모니터링 |
| **P2** | **Diagnostics 자동 덤프 번들** | 실패 복구 자료화 | 에러 발생 즉시 화면/HTML/콘솔을 덤프하여 완벽한 원인 진단 제공 |
| **P2** | **Drag and Drop JS Simulator** | 이미지 삽입 안정성 | 인풋 태그가 완전히 숨겨져 막히는 특수 상황에 대비한 최후 폴백 |
