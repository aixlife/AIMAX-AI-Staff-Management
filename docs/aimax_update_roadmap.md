# AIMAX 업데이트 로드맵

작성일: 2026-05-04

## 목적

이 문서는 AIMAX를 기존 설치형 프로그램에서 웹앱 중심 제품으로 전환하기 위한 장기 업데이트 지도를 정리한다. 실제 작업은 별도 MVP Phase 문서를 우선한다.

중요한 방향 전환:

- 2026-05-05에 확인된 핵심 문제는 사용자가 설치 파일을 실행하면 기존 Tkinter 전체 프로그램이 그대로 열려, 웹앱을 쓸 이유가 없어 보인다는 점이다.
- 따라서 다음 우선순위는 웹앱 UI 추가가 아니라 `웹앱 본체 + 화면 없는 Local Agent` 구조로 바로잡는 것이다.
- 새 세션에서는 반드시 `docs/local_agent_transition_handoff.md`를 먼저 읽고 Local Agent 전용화 Phase L0/L1부터 진행한다.

핵심 방향은 다음과 같다.

- 사용자는 웹앱에서 로그인하고 작업을 지시한다.
- 네이버 자동화는 사용자 PC의 로컬 실행기가 담당한다.
- 예리, 현주, 통합 상품은 하나의 코드베이스와 권한 시스템으로 구분한다.
- 오류 보고는 앱 안에서 수집하고 Oracle 서버로 모은다.
- 보안은 IP 차단이 아니라 계정 권한, 기기 등록, 실행 토큰으로 관리한다.

## 제품 원칙

1. 완전 무설치 SaaS를 1차 목표로 두지 않는다.
2. 네이버 로그인, 쿠키, Chrome 자동화는 사용자 PC에 남긴다.
3. 서버에는 네이버 비밀번호, 쿠키, 세션 원문을 저장하지 않는다.
4. 실행 파일 복사 방지는 설치 파일 보호보다 서버 권한 확인으로 해결한다.
5. 기존 구매자는 사용 흐름이 복잡해지지 않도록 웹앱 전환을 단계적으로 제공한다.
6. 업데이트와 오류 처리는 사용자가 카톡/단톡방에 설명하지 않아도 되도록 앱 내부에서 끝나게 만든다.

## 전체 업데이트 흐름

### Phase A. MVP 웹앱 전환 및 오류 보고

목적: 지금 구매한 사용자들이 프로그램이 아니라 웹앱에서 쓰는 경험을 갖게 한다.

범위:

- 웹앱 로그인
- 구매자/회원 계정 생성
- 로컬 실행기 연결 상태 표시
- 기존 자동화 기능 중 최소 핵심 기능 노출
- 오류 보고 버튼
- 로그/traceback/debug 목록 자동 수집
- Oracle 서버 `/api/reports` 전송
- 전송 실패 시 pending 저장
- 업데이트 안내 체계 초안

상세 실행 순서는 `docs/mvp_webapp_error_reporting_phase.md`를 기준으로 한다.

### Phase B. 상품 권한 및 기기 제한

목적: 예리, 현주, 통합 구매자를 하나의 코드와 권한 체계로 구분한다.

범위:

- 상품 코드: `yeri`, `hyunju`, `bundle`
- 권한별 메뉴 노출
- 계정당 등록 기기 제한
- 동시 실행 제한
- 관리자 기기 해제
- 권한 회수 시 다음 실행부터 기능 잠금

정책 초안:

- 예리: 예리 기능, 기기 1대
- 현주: 현주 기능, 기기 1대
- 통합: 예리 + 현주 + 통합 대시보드, 기기 1대 또는 2대
- IP 제한은 사용하지 않는다.

### Phase C. 운영자 관리자 콘솔

목적: 관리자가 DB를 직접 건드리지 않고 구매자와 오류를 운영한다.

범위:

- 이메일 검색
- 상품 권한 부여/회수
- 기기 등록/해제
- 오류 보고 목록
- 오류 상태 변경: `new`, `checking`, `fixed`, `need_user_info`
- 사용자에게 안내 메시지 발송
- pending report 재전송 상태 확인

초기에는 결제 자동화 없이 수동 권한 부여로 시작한다.

### Phase D. 회원가입 및 회원 인증 자동화

목적: 회원들이 일정 기간 직접 가입하고, 이후 인증 기반 자동 개통으로 전환한다.

초기:

- 회원가입 오픈 기간 운영
- 이메일 인증
- 가입자에게 베타/회원 권한 부여
- 관리자 화면에서 예리/현주/통합 권한 조정

구매자:

- 구매자 이메일을 로그인 ID로 사용
- 랜덤 임시 비밀번호 또는 1회용 초대 링크 발송
- 첫 로그인 시 비밀번호 변경 강제
- 초기 비밀번호를 이메일에서 예측 가능하게 만들지 않는다.

후기:

- 회원 인증 정보 입력
- 인증 통과 시 자동 권한 부여
- 인증 실패 시 관리자 검토 큐로 이동

### Phase E. 로컬 실행기 자동 업데이트

목적: 사용자가 새 파일을 찾아 다시 설치하지 않아도 안정적으로 최신 버전을 쓰게 한다.

범위:

- 실행기 시작 시 latest version 확인
- 업데이트 필요 시 웹앱/실행기에 안내
- 업데이트 다운로드
- 설치 후 재시작
- 웹앱은 서버 배포만으로 자동 갱신
- 로컬 실행기는 코드서명된 설치/업데이트 체계로 전환
- 다음 업데이트 후보: 신규 배포/업데이트는 OS별 통합 설치 파일 1개로 단순화한다. 예리/현주/통합을 별도 설치 파일 3개로 계속 빌드하면 Windows 재빌드와 운영 업로드가 병목이 되므로, 상품 구분은 설치 파일이 아니라 서버 권한과 Local Agent worker readiness로 처리한다. 기존 단독 설치 파일은 기존 링크 호환용 legacy로만 유지한다.

MVP에서는 자동 무음 업데이트보다 사용자 승인형 업데이트를 우선한다.
ZIP 다운로드는 기존 산출물을 웹앱에 빠르게 연결하기 위한 임시 방식으로만 사용했다. 최종 사용자 배포는 macOS `.dmg`, Windows `.exe` 설치파일을 내려받아 사용자가 직접 열어 실행하는 흐름으로 전환한다. 브라우저가 다운로드 파일을 자동 실행하는 방식은 보안상 지원하지 않는다.
2026-05-05 기준 macOS v1.0.1 실행기는 웹앱 연결 코드 포함 빌드, 운영 API E2E smoke test, DMG 설치파일 생성까지 완료했다. 테스트는 실제 웹앱 로그인/작업 생성/Agent 수신/상태 업데이트 경로를 사용하되, 네이버 실제 작업 함수는 더미로 대체했다. Windows는 별도 Windows 환경에서 v1.0.1 `.exe` 설치파일 빌드 후 업로드해야 한다.
웹앱 설정 UX는 `docs/webapp_settings_ux_phase.md`의 Guided Local Secure Setup 구조를 따른다. Windows 빌드는 readiness contract와 웹앱 정보구조를 확정한 뒤 진행한다.
2026-05-05 기준 Local Agent readiness contract, 서버 저장/조회, 웹앱 시작 체크리스트와 설정 탭 1차 구현, 웹앱의 “로컬 설정 열기” command 연결까지 완료했다. 운영 API smoke test에서 readiness 표시, `open_settings` command, 예리/현주 job dispatch가 통과했다. macOS DMG 3종도 새 Agent 코드로 재빌드/검증/업로드했다. Windows EXE 3종도 Windows 환경에서 빌드 완료 후 맥을 통해 운영 다운로드 폴더에 업로드했고, `/api/downloads/options`에서 Windows 3종 모두 `exists: true`를 확인했다.
2026-05-06 기준 L1J에서 GPT/OpenAI 모델, OpenAI 이미지 생성, 사용자 입력 OpenAI key 저장, 작은 headless 로컬 보안 설정 창, 글자수 ±5% 보정, usage token 기반 원화 비용 저장을 추가했다. 운영 웹앱과 macOS DMG 3종은 L1J 기준으로 재배포 완료했지만, 기존 Windows EXE 3종은 L1J 이전 파일이면 stale이므로 Windows 환경에서 다시 재빌드/업로드해야 한다. Windows 데스크탑 AI에게 전달할 상세 지시서는 `docs/windows_build_v1_0_1_handoff.md`, 짧은 실행 프롬프트는 `docs/windows_ai_build_prompt.md`에 준비했다.
기능 MVP 안정화 뒤에는 Lazyweb MCP를 활용한 UI Design Pass를 진행한다. Lazyweb는 실제 앱 화면/플로우 레퍼런스를 바탕으로 온보딩, 설정, 대시보드, 권한/상태 표시 UX를 다듬는 데 사용하고, 특정 앱 디자인을 그대로 복제하지 않는다.
2026-05-05 기준 현재 Codex 세션에는 Lazyweb MCP가 연결되어 있지 않아 공개 Lazyweb 레퍼런스 기반으로 1차 style brief를 작성했고, 대시보드 다음 행동/설정 단계형 레이아웃/작업 차단 notice를 운영 웹앱에 먼저 반영했다.
2026-05-05 기준 Lazyweb Codex 플러그인은 로컬 토큰 저장, marketplace 추가, `config.toml` 활성화까지 완료했다. MCP 직접 호출로 `tools/list`, `lazyweb_health`, `lazyweb_search {"query":"pricing page","limit":3}` 응답을 확인했다. 현재 실행 중인 Codex 세션은 새 플러그인 도구를 동적으로 로드하지 않으므로 Codex 재시작 후 네이티브 Lazyweb 도구 노출을 최종 확인하고 Phase 6 UI reference pass를 이어간다.
Codex 재시작 후 `codex mcp list`에서 `lazyweb` enabled를 확인했고, Lazyweb MCP 실제 검색 결과를 `docs/lazyweb_ui_style_brief.md`에 반영했다. 운영 웹앱에는 체크리스트 진행 index, 직원 상세 상태/CTA, 오류 보고 접수 ID 표시, 오류 보고 수집/보호/처리 안내를 추가 배포했다.
첫 로그인 비밀번호 변경 UX도 수정했다. 강제 변경 상태에서는 현재 비밀번호를 다시 요구하지 않고 새 비밀번호만 받으며, 비밀번호 관련 API 오류는 한국어로 안내한다. 로컬 smoke에서 임시 계정 생성, 로그인, 현재 비밀번호 없이 새 비밀번호 변경, 실행 가능 상태 전환을 확인했다.

### Phase F. 결제 자동화

목적: 반복 판매가 검증된 뒤 결제와 권한 부여를 자동화한다.

범위:

- 결제 완료 webhook
- 권한 자동 부여
- 환불/취소 시 권한 회수
- 상품 업그레이드
- 안내 메일 자동 발송

Phase A~D 이후 실제 구매 흐름이 안정되면 착수한다.

### Phase G. AIMAX 직원 확장

목적: AIMAX를 여러 업무 직원이 협업하는 구조로 확장한다. 직원은 실행 위치에 따라 로컬 직원, 서버 직원, 외부 AI 보조 직원으로 나눌 수 있다.

AI 직원 콘솔로 전환하는 실제 화면/서버 작업 순서는 `docs/aimax_ai_staff_console_phase.md`를 기준으로 진행한다. 이 문서는 Lazyweb MCP로 확인한 agent library, marketplace, team directory, automation dashboard 패턴을 AIMAX에 맞게 재구성한 실행 phase다.
2026-05-05 기준 Phase 1 서버 직원 카탈로그 정리를 시작해 `/api/workers` 응답을 확장했다. 예리/현주는 `staff_code`, `product`, `job_kind`, `required_settings`, 프로필 이미지, 설명, capability를 포함해 내려오며, 웹앱은 이 카탈로그를 읽어 기존 직원 메타데이터에 반영한다. 기존 readiness key와 job kind는 유지했으므로 macOS/Windows 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 2 웹앱 데이터 어댑터 정리도 운영 배포했다. 설정 체크리스트, 다음 행동, 직원 카드/상세 상태, 작업 폼 활성화 로직이 직원 카탈로그와 `required_settings`를 기준으로 계산된다. 작업 폼 UI는 아직 예리/현주 고정이며, 별도 직원 콘솔 UI는 Phase 3에서 진행한다. 실행기 계약은 그대로라 Windows 파일 변경은 필요 없다.
2026-05-05 기준 Phase 3 직원 콘솔 UI도 운영 배포했다. 좌측 내비게이션에 `직원` 탭을 추가하고, 직원 현황 요약, 검색, 상태 필터, 직원 카드 grid, 상세 패널, 최근 작업, 대표 CTA를 추가했다. 이는 웹앱 UI 변경만 포함하므로 Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 3.5 Visual Refresh도 운영 배포했다. 기존 navy/blue 중심의 평평한 관리자 화면에서 차콜/그린/뉴트럴 기반의 AI 직원 운영 콘솔 톤으로 색상, 패널, 버튼, 직원 카드, 상세 패널, 모바일 nav 스타일을 재정의했다. 웹앱 CSS/UI 변경만 포함하므로 Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 4 작업 화면 재구성도 운영 배포했다. 대시보드 하단의 작업 지시 폼을 `작업` 탭으로 이동하고, 직원 선택 카드와 선택 직원별 폼 표시 구조를 추가했다. 직원 상세/직원 탭의 `작업 지시하기` CTA는 해당 직원 작업 폼으로 이동한다. 기존 `/api/jobs`, job kind, readiness key는 유지해 Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 5 대시보드 재정의도 진행했다. 첫 화면에 운영 현황 카드, 막힌 항목, 빠른 이동을 추가해 사용 가능한 직원/필요 설정/진행 작업/실패 작업을 바로 볼 수 있게 했다. 작업 생성은 계속 `작업` 탭에 두며, 웹앱 UI 변경만 포함하므로 Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 6 설정/업데이트 분리도 진행했다. `설정` 탭은 로컬 설정과 실행기 연결 상태만 다루고, 새 `업데이트` 탭에서 현재 실행기 버전, 최신 버전, 필수 최소 버전, 추천 설치 파일, 권한별 설치 파일 목록을 확인하도록 나눴다. 필수 업데이트는 대시보드 막힌 항목에도 표시된다. 웹앱 UI 변경만 포함하므로 Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 7 오류 보고 운영화도 운영 배포했다. 서버는 오류 보고 저장 시 사용자 계정과 support 상태 필드를 함께 기록하고, 웹앱은 `GET /api/reports/mine`으로 로그인 사용자의 최근 오류 보고 목록과 처리 상태를 보여준다. 로컬 smoke에서 사용자 생성, 로그인, 오류 보고 생성, 내 보고 목록 조회까지 통과했고, 운영 `/health`, `/app`, `/api/reports/mine` 라우트 응답을 확인했다. Windows/macOS 실행기 재빌드는 필요 없다.
2026-05-05 기준 Phase 8A 레이아웃 운영성 개선도 진행했다. 데스크톱/노트북에서는 왼쪽 내비게이션을 고정하고 오른쪽 main 영역만 스크롤되도록 바꾸며, 체크리스트/막힌 항목/직원 목록/테이블은 내부 스크롤로 제한했다. 모바일에서는 고정 레이아웃을 풀어 자연스러운 세로 스크롤을 유지한다. 웹앱 CSS/UI 변경만 포함하므로 Windows/macOS 실행기 재빌드는 필요 없다.

범위:

- 로컬 직원: 네이버 로그인, Chrome/Selenium, 사용자 PC 파일 접근처럼 사용자 기기에서만 안전하게 실행해야 하는 역할
- 서버 직원: 계정, 권한, 작업 큐, 오류 보고, 업데이트, 알림, 운영 리포트처럼 중앙에서 처리해도 되는 역할
- 외부 AI 보조 직원: 리서치, 초안, 검수, 요약처럼 네이버 계정/쿠키 없이 처리 가능한 역할
- 예리: 소통글쓰기 중심 직원
- 현주: 영업/고객 찾기 중심 직원
- 추후 추가 직원: 검수, 정리, 예약, 리포트, 리서치 등
- 현재 MVP의 작업 코드는 `yeri_writer`, `hyunju_sales`만 둔다.
- 현재 MVP에서 네이버 자동화는 사용자 PC의 로컬 Agent가 실행한다.
- 서버는 우선 작업 큐, 권한, 상태, 오류 보고, 업데이트 확인을 담당한다.

명확히 제외:

- 사용자의 네이버 비밀번호, 쿠키, 세션 원문을 서버나 외부 AI로 보내는 구조
- 사용자의 맥북 밖에서 네이버 Chrome 자동화를 실행하는 구조
- MVP 단계에서 모든 직원/AI 역할을 한 번에 넣는 것

## Oracle 서버 운영 기준

현재 참고할 Oracle 연결 정보는 `.claude/memory/reference_oracle_personal_assistant.md`에 있다.

- SSH: `ssh -p 3333 ubuntu@100.69.85.89`
- OpenClaw Gateway: 서버 내부 `http://localhost:18789`
- 오류 보고 API는 Oracle 서버에 `/api/reports` 형태로 두는 것을 기본 가정으로 한다.

MVP 구현 시에는 서버 URL을 앱 설정값으로 분리한다.

예:

- 개발: 로컬 JSON 저장만 사용
- 테스트: `http://<oracle-host>:<port>/api/reports`
- 운영: HTTPS 도메인 연결 후 사용

## 보안 원칙

- 네이버 ID는 필요 시 일부 마스킹해서만 전송한다.
- 네이버 비밀번호는 절대 전송하지 않는다.
- API Key, 쿠키, 세션, 긴 토큰은 자동 마스킹한다.
- debug 파일은 1차에서는 파일 목록만 전송한다.
- debug 파일 내용 첨부는 별도 사용자 동의 후 처리한다.
- 서버는 원문 로그를 받더라도 저장 전 재마스킹을 한 번 더 수행한다.
- 실행기는 작업 실행 전 서버에서 짧은 실행 토큰을 받아야 한다.
- 계정 권한과 기기 등록이 맞지 않으면 작업을 실행하지 않는다.
