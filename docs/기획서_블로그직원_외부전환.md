# 기획서 — "블로그 직원" 컨셉 전환 및 외부 UI 재구성

**작성일**: 2026-04-23
**타깃 일정**: 2026-04-25 (토) 패밀리데이 행사
**기준 녹취**: `패밀리데이 행사 준비.txt`
**전제 원칙**: 내부 기능 로직은 그대로 유지. **외부적으로 보이는 "껍데기"만 교체**하여 행사 시연에 대응.

## 1. Executive Summary (한 장 요약)

- **방향 전환**: "블로그 자동화 프로그램"이 아니라 **"블로그 일을 대신 해주는 직원(가상 사원)"** 으로 브랜딩.
- **첫 직원 이름**: 회원 명단에서 랜덤 추출 (제안: **"블로거 예리"**).
- **UI 재구성**: 8개 분산 탭 → **4단계 워크플로우** (찾아올게요 → 친해질게요 → 설득할게요 → 판매할게요).
- **중복 제거**: 공감·댓글·공감+댓글 3개 기능을 1개로 통합, 대량발행 → 예약발행으로 재정의.
- **행사 목적**: 실제로 돌아가는 시연 + "직원이 대신 일한다"는 정서적 공감 획득.
- **명확한 선긋기**: 본 기획서는 **껍데기 개편(즉시 적용)** + **추가 기능 기획(문서화만, 추후 구현)** 으로 2단계 분리.

## 2. Why — 왜 지금 컨셉을 바꾸는가

### 2.1 기존 포지셔닝의 문제
- 현재 앱은 "네이버 블로그 자동화 도구" 느낌. 경쟁 도구와 차별성 없음.
- 회원들에게 "프로그램" 아이콘을 전달한다는 이미지는 약함 — 외부 어뷰징 툴과 구분이 안 됨.
- 실제로는 "직원 대신 쓰는 도구"인데, 표현이 그렇게 되어 있지 않음.

### 2.2 메이크패밀리 정체성 반영
- 회사 이름 "메이크패밀리" → 각 직원에게 **회원 이름을 부여**하면 브랜드 일치도 극대화.
- 패밀리데이 행사에서 "여러분 이름을 딴 직원을 선물해드립니다"는 서사 가능.
- 향후 직원 추가(인스타 직원 · 아나운서 직원 등)도 동일 구조로 확장.

### 2.3 행사 제약
- 25일(토) 패밀리데이 데모. 내일 모레.
- 내부 로직 변경은 시간 부족 + 위험. **가장 ROI 높은 것은 껍데기 교체**.

## 4. Phase 1 — 외부 UI 재구성 (즉시 적용, 25일 전 완료)

### 4.1 메인 구조: 4단계 워크플로우

```
┌─────────────────────────────────────────────┐
│  블로거 예리 (아바타)                       │
│  메이크패밀리의 첫 번째 블로그 직원          │
├─────────────────────────────────────────────┤
│                                             │
│  ① 고객을 찾아올게요   (= 찾는다 / 크롤링)  │
│  ② 고객과 친해질게요   (= 모은다 / 소통)    │
│  ③ 고객을 설득할게요   (= 쓴다 / 글쓰기)    │
│  ④ 고객에게 판매할게요 (= 판다 / TBD)       │
│                                             │
│  [설정] — 계정·API 키 (직원 설정으로 라벨)  │
└─────────────────────────────────────────────┘
```

### 4.2 각 단계별 매핑

#### ① 고객을 찾아올게요 (Find)
- **기존 기능**: `Scraper` (키워드로 블로거 ID + 이메일 수집)
- **표시 방식**: "블로거 예리가 키워드로 잠재 고객을 찾아옵니다"
- **선택지 2개**:
  - 검색어 기반 (현재 구현 완료)
  - 특정 네이버 아이디의 팔로워 수집 (**Phase 2 — 미구현, 기획만**)
- **결과**: CSV 다운로드 자동 → 다음 단계로 전달

#### ② 고객과 친해질게요 (Engage)
- **기존 기능**: `Like + Comment + Engage + Neighbor` 4개 → **2개로 통합**
  - (a) 소통하기 (공감 + 댓글) — `Engage` 유지
  - (b) 서로이웃 신청 — `Neighbor` 유지
- **삭제(숨김)**: `Like` 단독, `Comment` 단독 → 자주 안 쓰이고 `Engage`와 중복
- **표시 방식**: "예리가 공감하고 댓글 남기며 서로이웃 신청합니다"
- **기존 위험 고지 배너 유지** (서이추 차단 위험)

#### ③ 고객을 설득할게요 (Write)
- **기존 기능**: `Write` + `Bulk Post`
- **하위 탭으로 재구성**:
  - (a) 글 한 편 쓰기 (기존 Write)
  - (b) 예약 발행 (기존 Bulk Post을 재정의)
- **표시 방식**: "예리가 키워드와 소구점으로 글을 씁니다"
- **Phase 2 추가 계획**: 소구점(USP) 입력칸 (5장 참조)

#### ④ 고객에게 판매할게요 (Sell)
- **현재 기능 없음**. Placeholder로 생성만.
- **표시 방식**: "준비 중 — 다음 직원이 담당합니다"
- **패밀리데이 서사용**: "지금은 예리가 여기까지 할 수 있고, 다음 프로젝트에서 판매 담당 직원을 선물할 예정입니다"

### 4.3 비주얼 요소

| 요소 | 내용 |
|---|---|
| 아바타 | 단순 일러스트 원형 아바타 (무료 placeholder 이미지 1장) |
| 색상 | 메이크패밀리 브랜드 색상 유지 |
| 상단 타이틀 | `블로거 예리 · 메이크패밀리` |
| 서브타이틀 | 시간대별 인사 (예: "예리가 근무 중입니다 🟢") |
| 진행 상태 | "예리가 고객을 찾고 있습니다..." / "예리가 글을 쓰고 있습니다..." |

### 4.4 구현 범위 (Claude가 맡을 작업)

**변경 대상**: `app.py` 단일 파일 UI 레이어만.

- 좌측 사이드바 네비게이션 재그룹핑 (8개 → 4개 카테고리)
- 각 패널 타이틀·서브타이틀 문구 교체
- `Like`, `Comment` 패널 숨김 처리 (코드 삭제 X, 사이드바에서만 숨김)
- 진행 로그 문구 "예리가 ~합니다" 톤으로 통일
- 설정 탭 상단에 "직원 프로필" 카드 추가 (이름·역할·상태)

**변경 안 하는 부분**: 워커 함수, 엔진 모듈, 빌드 설정.

## 5. Phase 2 — 추가 기능 기획 (문서화, 추후 구현)

**원칙**: 기획 승인 후 별도 스프린트에서 구현. 이번 행사에는 포함 안 함.

### 5.1 서이추 멘트 AI 개인화 ⭐ (회의 중 명시적 요청)
- **문제**: 현재는 사용자가 직접 작성한 멘트 풀 중 랜덤 선택
- **개선**: 사용자 프로필(대표 소개·제품 소구점)을 받아 AI가 멘트 10개를 자동 생성
- **입력 예시**: "저는 경제적 자유를 높이는 대표입니다. 돈 버는 수익 정보를 드립니다"
- **출력**: 수신자가 "받아줄 이유가 있는" 맞춤 멘트 10종
- **기술**: 기존 Gemini/Claude API 재사용
- **우선순위**: 높음 (서이추 수락률 직결)

### 5.2 글쓰기 소구점(USP) 주입 ⭐ (회의 중 가장 강조된 부분)
- **문제**: 현재 키워드만 입력 → AI가 일반 정보성 글 작성 → "왜 메이크패밀리가 만들었는지" 흔적 없음
- **개선**: 글쓰기 탭에 "제품/서비스 소구점 3개" 입력칸 추가
- **동작**: AI가 키워드 + 소구점을 함께 받아 자연스럽게 녹여 작성
- **핵심 논리**: "이 직원은 단순 글쓰기가 아니라, 여러분 서비스의 소구점을 가진 글을 씁니다"
- **우선순위**: 최상 (이거 없으면 차별화 없음 — 회의에서 가장 많이 지적됨)

### 5.3 팔로워 크롤링 (네이버 ID 기반)
- **현재**: 검색어로 블로거 ID 수집
- **추가**: 특정 네이버 아이디를 입력 → 그 아이디의 팔로워 목록 수집
- **난이도**: 중 (네이버 블로그 이웃 페이지 구조 분석 필요)
- **우선순위**: 중

### 5.4 예약 발행 (대량발행 재정의)
- **현재**: `Bulk Post` — 엑셀로 여러 계정 순차 실행. 용도 모호.
- **개선**: 한 계정 내에서 글 N개를 시간 간격 두고 자동 발행
- **예시**: "23일 오전 10시, 24일 오후 2시, 25일 오후 6시에 각 1개씩"
- **제약**: 컴퓨터가 켜져 있어야 함 (사용자 안내 문구 필수 추가)
- **우선순위**: 중

### 5.5 공감·댓글 변칙 멘트
- **문제**: 댓글 멘트가 고정 패턴 → 봇 탐지 위험 ↑
- **개선**: 대상자 블로그 최근 글을 읽고 맥락 기반 댓글 AI 생성 (이미 부분 구현됨)
- **추가**: 사용자 프로필 맞춤 멘트 자동 생성 (서이추 멘트와 동일 로직)
- **우선순위**: 중

## 7. Claude의 독립적 의견 (Professional Honesty)

> 회의에서 직접 나온 방향을 그대로 따르지 않고, **빠질 수 있는 함정**과 **보완 포인트**를 정리합니다.

### 7.1 "껍데기만 바꾸기" 전략의 가장 큰 리스크
- **행사장에서 한 번이라도 에러/크래시 나면 컨셉 전체가 붕괴**합니다.
- "예리가 고객을 찾아올게요" 클릭 → 에러 팝업 → 관객은 "아 그냥 미완성 툴이네" 인식.
- **필수 방지책**:
  1. 25일 오전까지 **전체 4단계 흐름 리허설 최소 2회**
  2. 실제 네이버 계정(테스트용) 기준 end-to-end 검증
  3. **녹화 데모 백업 영상** 준비 — 라이브 실패 시 즉시 전환
- 이것 없이 껍데기만 바꾸면 "포장만 그럴싸한 미완성품"이 되어 역효과.

### 7.2 "직원 만들기" 컨셉의 정의 공백
- 회의에서 "직원이다", "의인화한다"는 합의는 명확. **그러나 "직원"의 범위·규칙이 정의 안 됨**.
- 미정의 지점들:
  - 직원은 **하나의 계정만 운영**? 아니면 여러 계정?
  - 직원은 **하나의 플랫폼**? 아니면 크로스 플랫폼?
  - 직원 해고/교체는 어떻게?
  - 여러 사용자가 같은 "예리"를 공유? 각자 다른 예리?
- **권장**: 25일 전에 단 한 문장으로 정의 고정. 예: "직원 1명 = 1개 플랫폼 + 1개 계정 + 1개 페르소나"
- 정의 없으면 다음 직원(인스타 담당) 만들 때 같은 논쟁 반복됨.

### 7.3 회원 이름 사용의 법적·도의적 검토
- "예리님 이름을 딴 직원을 선물"은 감동 포인트지만 리스크 존재:
  - 회원 동의 없이 이름만 가져다 쓰는 건 **개인정보 이슈** 소지
  - 해당 회원이 퇴회/이탈 시 직원 이름도 바꿔야 하는 번거로움
  - 다른 회원이 "왜 내 이름은 안 쓰냐" 형평성 문제
- **권장**:
  - 이름 차용 **사전 동의 문구 1장** 준비 (행사 현장에서 서명 또는 구두 동의)

### 7.4 서이추 자동화 + 직원 브랜딩의 충돌
