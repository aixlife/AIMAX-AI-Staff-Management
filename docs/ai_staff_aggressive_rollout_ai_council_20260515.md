# AIMAX 신규 직원 공격 배포 일정

작성일: 2026-05-15  
용도: 동규 대표 공유 및 Notion 일정 등록용  
전제: 신규 직원 5명은 이미 사용자 화면에 카드로 노출되어 있음. 이제 목표는 "보이게 하기"가 아니라 "최소 기능을 실제로 쓰게 하기"다.

## 1. AI Council 실행 방식

보안상 실제 서비스명, 직원명, 레포 URL, 내부 경로는 외부 모델에 전달하지 않았다. 아래의 익명화된 모듈 유형만 전달했다.

- AI 콘텐츠/스크립트 생성
- 자료 조사/레퍼런스 정리
- 프로젝트/작업 관리
- 회계/매출 비용 대시보드
- 로컬 데스크톱 판서 앱

Council 결과 위치:

```text
council-runs/20260515-084953-a-saas-product-already-shows-five-upcoming-featu/
```

Council 공통 의견:
- AI 콘텐츠 생성과 자료조사는 같은 파이프라인으로 묶어 병렬 개발한다.
- PM은 별도 운영 트랙으로 진행한다.
- 회계는 권한, 백업, 감사 로그가 먼저다.
- 데스크톱 앱은 웹 기능이 아니라 패키징/버전/배포 리스크가 핵심이다.
- AI 비용 가드는 첫날부터 붙여야 한다.

## 2. 최종 판단

일반 개발팀 일정이면 1주 1명이 맞지만, 지금 상황은 다르다.

이미 완료된 것:
- 사용자 화면에 신규 직원 5명 카드 노출
- 직원별 이미지 등록
- 레포별 껍데기 확인
- 통합 방향 문서화

따라서 AI 개발 속도 기준으로는 2026-05-29까지 신규 5명 전원을 "제한 베타로 최소 사용 가능" 상태까지 여는 공격 일정이 가능하다.

단, 여기서 말하는 "사용 가능"은 완성형이 아니다. 각 직원마다 핵심 기능 1개만 먼저 열고, 나머지는 이후 고도화한다.

## 3. 대표 공유용 한 줄

2026-05-18에 직원 운영 기반을 먼저 배포하고, 2026-05-20부터 윤미/송이를 시작으로 2026-05-29까지 신규 5명 모두 제한 베타로 실제 사용 가능하게 여는 일정으로 가겠습니다. 완성형은 아니지만, 각 직원의 핵심 기능을 먼저 열고 비용/권한/로그 안전장치를 붙인 상태로 순차 오픈하겠습니다.

대표/Notion용 쉬운 단계명:

| 개발 용어 | 쉬운 표현 | 의미 |
| --- | --- | --- |
| 내부 알파 | 회사에서 사용해보기 | 우리 팀/대표가 먼저 써보고 큰 문제를 잡는 단계 |
| 제한 베타 | 기존 사용자들에게 배포 | 이미 쓰는 고객 일부 또는 전체에게 조심스럽게 여는 단계 |
| 배포 후보/RC | 외부 판매 준비 | 신규 고객에게 팔아도 될지 마지막으로 확인하는 단계 |

## 6. 병렬 개발 전략

### 트랙 A: 직원 운영 기반

기간:
- 2026-05-15 ~ 2026-05-18

목표:
- 직원 카탈로그 데이터화
- 관리자 직원 관리
- 상태/노출/권한 제어
- 실행 가능 직원과 준비 중 직원 분리

이 트랙은 모든 직원 배포의 선행 조건이다.

### 트랙 B: 윤미 + 송이

기간:
- 2026-05-18 ~ 2026-05-25

목표:
- 자료조사와 스크립트 생성을 하나의 콘텐츠 파이프라인으로 연결한다.

MVP 컷:
- 자동 크롤링 제외
- 복잡한 프로젝트 폴더 제외
- 입력은 URL/텍스트/메모 중심
- 결과는 저장/복사 중심

필수 안전장치:
- AI 호출 비용 확인
- 사용자별 일일 호출 제한
- 자동 유료 재시도 금지
- 요청 ID와 오류 로그 저장

### 트랙 C: 현성

기간:
- 2026-05-21 ~ 2026-05-27

목표:
- 내부 운영 PM으로 시작한다.

MVP 컷:
- Kanban/상태 보드 중심
- 알림, 캘린더 연동, 복잡한 권한 제외
- 직원별 대기/실패/진행 작업 요약 우선

### 트랙 D: 상수

기간:
- 2026-05-22 ~ 2026-06-12

목표:
- 회계 직원은 제한 베타까지 빠르게 열되, 일반 사용 후보는 늦게 판단한다.

MVP 컷:
- 수동 입력
- 합계/간단 손익
- CSV 내보내기
- 샘플 데이터 모드

필수 안전장치:
- 권한 구분
- 백업/내보내기
- 수정/삭제 감사 로그
- 관리자/지정 사용자 제한

### 트랙 E: 나경

기간:
- 2026-05-22 ~ 2026-06-12

목표:
- 웹 통합이 아니라 로컬 앱 파일럿으로 연다.

MVP 컷:
- 자동 업데이트 제외
- 실시간 웹 연동 제외
- 계정 페어링 제외
- 수동 다운로드와 버전 표시 중심

필수 안전장치:
- 다운로드 파일 버전 관리
- 앱 실행/설치 안내
- OS 권한 이슈 안내

## 7. 일정 리스크와 대응

AI 비용:
- 윤미/송이는 사용자당 일일 호출 제한을 둔다.
- 실패 시 자동 재시도하지 않는다.
- 요청 ID로 기존 결과를 조회하는 흐름을 만든다.

회계 데이터:
- 상수는 일반 사용자 전체 오픈을 서두르지 않는다.
- 제한 베타는 관리자/지정 사용자만 허용한다.
- 백업과 감사 로그 없이는 RC로 올리지 않는다.

로컬 앱:
- 나경은 "웹 기능 완성"이 아니라 "수동 설치 파일럿"으로 정의한다.
- 자동 업데이트를 첫 버전에 넣지 않는다.

기대치:
- 사용자에게 "정식 직원"이 아니라 "베타 직원"임을 명확히 표시한다.
- 직원별 첫 기능은 하나만 제공한다.
