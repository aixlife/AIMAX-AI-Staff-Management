# AIMAX 최종 구조 의견: Mac/Windows 공통 AI 직원 시스템

- 작성일: 2026-05-23
- 목적: 교차검증용 최종 설계 의견 정리
- 범위: AIMAX 웹앱, 서버, macOS/Windows 로컬 실행기, AI 직원 UX
- 핵심 목표: 사용자가 원하는 대로 직원이 실제로 끝까지 작동하는 단순하고 안정적인 제품 구조

## 1. 최종 결론

AIMAX는 Mac/Windows 어디서든 **같은 직원이, 같은 방식으로, 예측 가능하게 일하는 서비스**가 되어야 한다.

지금부터의 기준은 "직원이 많아 보이는 것"이 아니라 **각 직원이 자기 역할을 끝까지 완수하는 것**이다. 사용자는 내부 구조를 이해하지 않아도 되어야 하며, 웹앱만 보면 다음을 알 수 있어야 한다.

- 이 직원이 지금 사용 가능한지
- 어떤 설정이 부족한지
- 지금 어느 단계에서 일하고 있는지
- 실패했다면 어디서 멈췄는지
- 다시 시도하면 어디서부터 이어지는지
- 추가 비용이 드는지

따라서 구조는 "기능 추가"보다 "직원별 완성된 업무 경험"을 중심으로 다시 잡아야 한다.

## 2. 제품 원칙

### 2.1 웹앱이 지휘본부다

웹앱은 사용자의 중심 화면이어야 한다.

웹에서 담당해야 하는 것:

- 로그인
- 직원 선택
- 설정 상태 확인
- 작업 입력
- 작업 시작
- 진행 상태 확인
- 결과 확인
- 오류 보고
- 비용/재시도 안내

사용자는 웹앱만 보고 현재 상태를 이해할 수 있어야 한다.

### 2.2 로컬 실행기는 필요한 일만 한다

로컬 실행기는 웹이 할 수 없는 일만 담당해야 한다.

로컬 실행기가 담당할 일:

- 네이버 브라우저 자동화
- 네이버 로그인/session 유지
- OS 보안 저장소 접근
- 로컬 브라우저/권한/설치 상태 보고

로컬 실행기가 담당하지 말아야 할 일:

- 송이/윤미 같은 web-first 직원 처리
- 일반 AI/API 키 관리의 주 저장소 역할
- 서버에서 처리 가능한 분석/생성 작업
- 직원 정의/권한 판단의 최종 기준

### 2.3 Mac/Windows는 같은 코어, 다른 어댑터다

macOS와 Windows는 같은 직원 로직을 사용해야 한다. 다른 부분만 플랫폼 adapter로 분리한다.

공통 코어:

- agent heartbeat
- command polling
- job execution protocol
- browser automation interface
- error/report payload
- version/build id reporting

플랫폼 adapter:

- macOS Keychain
- Windows Credential Manager
- macOS installer/update
- Windows installer/update
- browser path detection
- OS 권한 팝업/보안 정책

`app.py`와 `split_version/app.py`처럼 같은 로직이 여러 벌로 갈라지는 구조는 장기적으로 제거해야 한다.

## 3. 직원별 실행 방식

모든 직원은 처음부터 다음 셋 중 하나로 분류해야 한다.

- `web-first`: 서버/웹만으로 실행 가능
- `local-agent-required`: 로컬 OS/브라우저 자동화가 핵심
- `hybrid`: 서버가 생성/분석하고 로컬 실행기가 최종 브라우저 자동화만 수행

### 3.1 예리

권장 방식: `hybrid`

예리의 최종 흐름:

```text
웹/서버:
키워드 입력
→ AI 글 생성
→ 이미지 생성
→ 결과 artifact 저장

로컬 실행기:
네이버 로그인
→ 저장된 글/이미지 가져오기
→ 네이버 에디터 입력
→ 임시저장/발행
```

이 구조가 필요한 이유:

- 글 생성과 네이버 입력 실패를 분리할 수 있다.
- 네이버 입력만 실패했을 때 AI 비용을 다시 쓰지 않고 이어갈 수 있다.
- API 키 저장 위치를 웹으로 통일할 수 있다.
- Mac/Windows 차이는 네이버 브라우저 자동화 단계로 제한된다.

### 3.2 현주

권장 방식: `local-agent-required`

현주는 네이버 기반 서로이웃/영업 자동화가 핵심이므로 로컬 실행기가 필요하다.

단, 웹은 다음을 명확히 보여줘야 한다.

- 네이버 계정 준비 여부
- 서로이웃 멘트 준비 여부
- 실행기 연결 여부
- 작업 속도/한도
- 실패 단계

### 3.3 송이

권장 방식: `web-first`

송이는 자료조사원이므로 실행기를 요구하지 않는 것이 맞다.

송이 흐름:

```text
프로젝트 생성/선택
→ 링크 입력
→ 수집/분석 진행
→ 결과 카드
→ 상세 브리프
→ 다른 직원에게 재활용
```

Apify 토큰은 웹에서 사용자별 암호화 저장 후 서버에서 사용한다.

### 3.4 윤미

권장 방식: `web-first`

윤미는 스크립트 생성 직원이므로 서버/웹에서 처리하는 것이 맞다.

필요한 개선:

- 관리자에서 윤미 권한을 켜고 끌 수 있어야 한다.
- hardcoded whitelist 방식은 제거해야 한다.
- bundle 권한과 윤미 개별 권한의 관계를 명확히 해야 한다.

### 3.5 나경/현성/상수

권장 방식: 아직 노출 보류

실제 기능이 없으면 일반 사용자에게 직원처럼 노출하면 안 된다.

가능한 노출 방식:

- 운영자 전용
- 준비 중 영역
- 베타 신청 영역
- 실제 job kind와 처리 흐름이 생긴 뒤 일반 노출

## 4. 사용자 흐름

최종 사용자 흐름은 단순해야 한다.

```text
1. 로그인
2. 직원 선택
3. 필요한 설정만 확인
4. 작업 입력
5. 시작
6. 진행 상태 보기
7. 결과 확인
8. 실패하면 이어서 재시도
```

사용자가 보면 안 되는 것:

- 내부 stacktrace
- 영어 중심 에러
- raw error code
- 서버/로컬 저장소 차이
- 불필요한 API 키 입력칸
- 아직 못 쓰는 직원 카드
- Mac/Windows 빌드 내부 사정

사용자가 봐야 하는 것:

- 이 직원은 지금 사용 가능함
- 이 직원은 실행기 연결 필요함
- 이 설정이 부족함
- 지금 이 단계에서 작업 중임
- 여기서 실패했음
- 다시 시도하면 어디서부터 이어짐
- 추가 비용이 드는지 안 드는지

## 5. UI/UX 방향

### 5.1 메인 화면

메인 화면은 기능을 많이 보여주는 대시보드보다, 지금 사용 가능한 직원과 상태를 명확히 보여주는 화면이어야 한다.

상단 상태:

- 실행기 연결됨/미연결
- 필요한 업데이트 있음/없음
- AI/API 연결 상태
- 최근 오류/확인 필요 상태

중앙:

- 사용 가능한 직원만 표시
- 준비 중 직원은 일반 직원과 같은 비중으로 노출하지 않음

직원 클릭 시:

- 작업 입력 폼
- 시작 버튼
- 최근 결과
- 필요한 설정 안내
- 이어서 재시도 버튼

### 5.2 꼭 필요한 버튼

버튼은 다음처럼 최소화한다.

- `작업 시작`
- `이어서 다시 시도`
- `결과 보기`
- `실행기 연결`
- `네이버 설정`
- `AI/API 연결`
- `오류 보고`

나머지는 숨기거나 고급 설정으로 이동한다.

### 5.3 설정 화면

설정은 세 영역으로 분리한다.

```text
AI/API 연결
- Gemini
- OpenAI
- Claude
- Apify

네이버 연결
- 네이버 ID
- 네이버 비밀번호
- 로그인 상태 확인

실행기 상태
- macOS/Windows
- 현재 버전
- 최신 버전
- 마지막 연결 시간
- 업데이트 버튼
```

핵심 원칙:

- AI/API 키는 웹에서 관리한다.
- 네이버 계정은 로컬 실행기에서 관리한다.
- 기존 로컬 API 키는 사용자 동의 후 1회 웹으로 가져온다.
- 원문 key는 저장 후 다시 보여주지 않는다.

## 6. 실패 처리 기준

오류는 사용자가 이해할 수 있는 업무 상태로 보여줘야 한다.

예리 실패 예시:

```text
예리 작업이 멈췄습니다.

완료된 단계
- 글 생성 완료
- 이미지 생성 완료

멈춘 단계
- 네이버 에디터 입력

다시 시도하면
- 생성된 글과 이미지를 재사용합니다.
- 추가 AI 비용 없이 네이버 입력 단계부터 이어갑니다.
```

오류 보고에는 다음을 포함한다.

- 직원명
- 작업 종류
- 실패 단계
- 사용자에게 보인 한국어 메시지
- job id 또는 command id
- 실행기 플랫폼
- 실행기 버전
- build id
- credential source 상태

절대 포함하지 않을 것:

- API key
- token
- password
- cookie
- signed URL
- raw private log

## 7. 기술 구조

권장 구조:

```text
Server
- auth
- users
- employees
- jobs
- secrets
- agents
- reports
- research
- billing/cost

Web
- employee catalog
- settings
- job runner
- result viewer
- error assistant

Local Agent Core
- agent heartbeat
- command polling
- job execution
- browser automation interface

Platform Adapter
- macOS keychain
- Windows credential manager
- macOS installer/update
- Windows installer/update
- browser path detection
```

저장소:

- JSON 파일 저장은 장기 운영 기준으로 부적합하다.
- 최소 SQLite WAL, 가능하면 Postgres로 이전한다.
- jobs, commands, reports, users, user secrets는 트랜잭션 저장소가 필요하다.

릴리스:

- Mac/Windows 모두 build id를 보고해야 한다.
- 웹 업데이트 화면은 서버 manifest, 다운로드 파일, 실행 중인 버전이 일치하는지 보여줘야 한다.
- 수작업 handoff/zip 전달은 장기적으로 Git 기반 릴리스 파이프라인으로 바꿔야 한다.

## 8. 실행 순서

### Phase 0. 현재 운영 안정화

목표:

- 사용자가 바로 실패하는 흐름을 줄인다.

작업:

- 예리 API 키 혼선 차단
- 로컬 설정 무한로딩 방지
- Mac/Windows 실행기 버전 불일치 표시
- 오류 메시지 한국어화
- 작업 실패 시 이어서 재시도 기준 표시

검증:

- 유료 API 없이 설정/상태/오류 흐름 smoke
- 네이버 실제 발행 없이 자동화 단계 dry-run
- macOS/Windows 각각 같은 화면과 같은 상태 문구 확인

### Phase 1. 직원 registry 단일화

목표:

- 직원 표시, 권한, 실행 방식, 필요한 설정을 하나의 기준으로 통일한다.

작업:

- 서버 employee registry 확정
- 웹 하드코딩 직원 목록 축소
- 관리자 권한 토글 정리
- 미완성 직원 일반 노출 제한

검증:

- 권한별 보이는 직원이 정확함
- 준비 중 직원은 실행 가능한 직원처럼 보이지 않음
- Mac/Windows에서 같은 계정으로 같은 직원 상태가 보임

### Phase 2. 비밀값 구조 확정

목표:

- 사용자가 어디에 무엇을 입력해야 하는지 헷갈리지 않게 한다.

작업:

- AI/API 키는 웹 저장으로 통일
- 네이버 계정은 로컬 저장으로 통일
- 기존 로컬 API 키 import 제공
- fallback 저장 안내와 제한 명확화

검증:

- 웹 저장 키가 저장 상태로 표시됨
- 로컬 설정 화면에 불필요한 provider key 입력란이 없음
- keychain 불가 환경에서도 안내와 fallback 동작이 명확함

### Phase 3. 예리 hybrid 완성

목표:

- 예리가 진짜 블로그 직원처럼 끝까지 일하게 한다.

작업:

- 서버 글/이미지 생성
- artifact 저장
- 로컬 네이버 입력
- 실패 단계별 재시도
- 비용 중복 방지

검증:

- 글 생성 성공 후 네이버 입력 실패 시 추가 AI 비용 없이 재시도
- Mac/Windows 모두 같은 job artifact를 입력 가능
- 사용자에게 단계별 상태가 한국어로 표시됨

### Phase 4. 송이/윤미 web-first 완성

목표:

- 실행기가 필요 없는 직원은 웹에서 끝까지 작동하게 한다.

작업:

- 송이 Apify/API 토큰 웹 저장 사용
- 송이 프로젝트/링크/브리프 흐름 완성
- 윤미 관리자 권한/작업 흐름 완성
- 결과를 다른 직원이 재활용할 수 있는 구조화 output으로 저장

검증:

- 송이는 실행기 없이 사용 가능
- 윤미는 권한 부여 후 바로 사용 가능
- 결과가 예리/윤미 등 다른 직원 입력으로 재사용 가능

### Phase 5. DB/릴리스/관측성 개편

목표:

- 업데이트할수록 안정성이 떨어지는 구조를 끝낸다.

작업:

- SQLite WAL 또는 Postgres 이전
- release manifest 도입
- build id 보고
- Mac/Windows smoke matrix
- job trace/report dashboard
- `app.py`/`split_version/app.py` 중복 제거

검증:

- 배포 전 no-paid smoke 통과
- 배포 전 no-Naver-write smoke 통과
- Mac/Windows 설치/실행/업데이트 상태 일치
- 대표 오류 보고 케이스 회귀 테스트 통과

## 9. 교차검증 질문

다른 AI 또는 개발자가 검토할 때 아래 질문에 답해야 한다.

1. 예리를 `hybrid`로 두는 판단이 맞는가?
2. 송이/윤미를 `web-first`로 두는 판단이 맞는가?
3. AI/API 키는 웹 저장, 네이버 계정은 로컬 저장으로 분리하는 것이 보안/UX 관점에서 맞는가?
4. 기존 로컬 API 키를 웹으로 import할 때 사용자 동의와 실패 복구 설계가 충분한가?
5. Mac/Windows 공통 코어 + 플랫폼 adapter 구조가 현재 코드에서 현실적인가?
6. `app.py`와 `split_version/app.py` 중복 제거를 어느 단계에서 해야 하는가?
7. JSON에서 SQLite WAL로 먼저 갈지, 바로 Postgres로 갈지 운영 규모상 무엇이 맞는가?
8. 미완성 직원을 숨기는 것이 판매/마케팅과 제품 신뢰 사이에서 맞는 균형인가?
9. 예리 실패 재시도 시 비용 중복 방지 설계가 충분한가?
10. 사용자가 보는 UI에서 더 제거해야 할 버튼이나 정보가 있는가?

## 10. 최종 판단

AIMAX는 방향이 좋다. "AI 직원"이라는 제품 컨셉도 강하다. 하지만 지금부터는 직원 수를 늘리는 것이 아니라, **한 명의 직원이 진짜 직원처럼 끝까지 일하는 기준**을 먼저 만들어야 한다.

우선순위는 다음이다.

1. 예리 하나를 Mac/Windows에서 안정적으로 완성한다.
2. API 키와 네이버 설정 경계를 확정한다.
3. 로컬 실행기 공통 구조를 정리한다.
4. 직원 registry를 단일화한다.
5. 송이와 윤미를 web-first로 완성한다.
6. 그 다음에 새 직원을 추가한다.

이 기준이 만들어지면 AIMAX는 단순 자동화 도구가 아니라, 사용자가 실제로 믿고 맡길 수 있는 AI 직원 시스템이 될 수 있다.
