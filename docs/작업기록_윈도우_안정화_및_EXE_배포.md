# NaverBlogAuto 작업 기록

## 문서 목적

이 문서는 이번 작업에서

- 어디서부터 시작했는지
- 어떤 문제가 있었는지
- 어떤 기준으로 원인을 좁혀 갔는지
- 어떤 시도를 했고 무엇을 채택/보류했는지
- 최종적으로 무엇을 완료했는지

를 한 번에 이해할 수 있도록 남긴 기록이다.

작성 기준은 "실제로 확인한 사실" 우선이다.  
대화 중에 나온 아이디어와 가설도 적되, 최종 반영 여부를 분리해서 적었다.

## 1. 시작점

### 작업 목표

이번 작업의 목표는 두 가지였다.

1. 네이버 로그인부터 블로그 글 발행까지 Windows에서도 안정적으로 동작하게 만들기
2. 개발 환경이 없는 Windows PC에서도 실행할 수 있도록 `exe` 배포물을 만들기

### 작업 대상 폴더

초기 기준 폴더는 사용자가 "맥북에서 성공한 버전"이라고 알려준 소스였다.

- 원본 참고 폴더: `C:\Users\likim\Desktop\NaverBlogAuto-main`

이후 원본을 더럽히지 않기 위해 별도 복제본을 만들어 작업했다.

- Windows 점검/수정 복제본: `C:\Users\likim\Desktop\NaverBlogAuto-main-wincheck`

즉, 실제 수정과 검증은 모두 `NaverBlogAuto-main-wincheck`에서 진행했다.

## 2. 처음 마주한 문제

작업 초반에 확인된 핵심 문제는 다음과 같았다.

### 2.1 로그인 페이지 구조 문제

네이버 로그인 기본 URL이 QR 중심 페이지로 바뀌면서, 예전처럼 바로 `#id`, `#pw` 필드를 기대하면 로그인 폼 자체가 없는 경우가 생겼다.

증상:

- 로그인 시도 단계에서 입력 필드를 못 찾음
- 자동 로그인 로직이 시작도 못 함

핵심 원인:

- 로그인 URL이 `https://nid.naver.com/nidlogin.login` 그대로여서 폼 모드가 강제되지 않음

### 2.2 세션은 있는데 블로그 쪽에서 로그인으로 인식하지 못하는 문제

`www.naver.com`에서는 로그인처럼 보이는데 `blog.naver.com` 또는 발행/댓글 흐름에서는 로그아웃처럼 동작하는 케이스가 있었다.

증상:

- 메인 네이버 로그인 상태와 블로그 로그인 상태가 분리됨
- 글쓰기 진입 시 NID 로그인 페이지로 다시 튐
- 같은 계정인데도 블로그 도메인에서 재인증을 요구함

### 2.3 에디터 iframe 진입 타이밍 문제

블로그 글쓰기 화면에 들어가도 Smart Editor가 아직 다 뜨기 전에 코드가 너무 빨리 판단해서 실패하는 문제가 있었다.

증상:

- `mainFrame` 안으로 들어간 직후 곧바로 실패
- 실제로는 잠시 기다리면 에디터가 뜨는데 코드가 먼저 포기
- "글쓰기 에디터를 찾지 못했습니다" 류의 오류

### 2.4 Windows 환경 의존성 문제

맥북에서 되던 코드가 Windows에서 그대로 항상 되는 건 아니었다.

확인 포인트:

- 브라우저 실행 파일 위치가 PC마다 다름
- 세션/로그/프로필 저장 경로를 현재 폴더에 두면 권한/배포 문제 발생 가능
- PyInstaller로 만든 결과물이 단일 exe가 아니라 `onedir` 구조인데, 문서상으로는 exe 하나만 쓰는 것처럼 보일 수 있음

### 2.5 배포 단계의 Windows 정책 문제

배포물 자체가 빌드되더라도 일부 Windows 환경에서는 서명되지 않은 exe를 정책으로 막을 수 있다.

이건 코드 버그와 다른 종류의 문제다.

## 3. 문제를 푼 방식

이번 작업은 무작정 셀렉터 하나씩 고치는 방식보다는, "네이버가 자동화를 막는다면 어디를 볼까?"를 역으로 생각하는 방식으로 진행했다.

### 3.1 역설계 관점에서 본 체크 포인트

네이버 입장에서 자동화를 막으려면 보통 아래를 본다고 가정했다.

1. 로그인 페이지 진입 형태가 비정상적인가
2. 입력 방식이 너무 기계적인가
3. 메인 도메인과 블로그 도메인 세션 연결이 어긋나는가
4. 글쓰기 iframe과 실제 에디터 DOM 로드가 완료되기 전에 자동화가 먼저 움직이는가
5. 브라우저 프로필/쿠키/경로가 환경마다 흔들리는가
6. 배포 시 파일 구조나 권한 문제로 실행이 깨지는가

이 관점으로 문제를 나누면 "로그인 자체", "세션 복원", "에디터 진입", "배포/실행" 네 축으로 분리할 수 있었고, 실제로도 이 축으로 정리했을 때 해결이 빨랐다.

### 3.2 웹 검색 대신 먼저 한 일

사용자가 필요하면 웹 검색도 하자고 했지만, 이번 건은 먼저 로컬 코드와 실제 실행 로그만으로도 결정적인 단서가 충분했다.

실제로는 아래 순서가 더 효과적이었다.

1. 맥북에서 성공했다는 원본 폴더를 기준점으로 삼음
2. Windows용 복제본을 따로 만들어 안전하게 수정
3. 로그인, 세션 복원, 글쓰기 진입, 발행을 각각 짧은 스모크 테스트로 분리
4. 실패 시 HTML/DOM/로그를 저장해서 "어디까지 왔는지"를 확인

즉, 이번 작업은 웹 검색보다 "현재 코드와 현재 네이버 페이지에서 실제로 무슨 일이 일어나는지"를 확인하는 쪽이 더 중요했다.

## 4. 검토한 아이디어와 판단

대화 중 여러 아이디어가 나왔고, 그중 일부는 채택했고 일부는 보류했다.

### 4.1 채택한 방향

- 로그인 URL을 폼 모드로 고정
- 세션 저장/복원 강화
- 블로그 글쓰기 진입 후 iframe/에디터 준비 대기 강화
- 브라우저 실행 파일 탐지 로직을 Windows 친화적으로 확장
- 실행 중 생성되는 데이터는 `%APPDATA%\NaverBlogAuto`로 일원화
- 배포물은 `exe` 단품이 아니라 폴더형 + zip 기준으로 정리

### 4.2 검토했지만 최종 핵심 해결책으로 채택하지 않은 방향

#### Scrapling 도입

사용자가 [Scrapling](https://github.com/D4Vinci/Scrapling.git)도 참고해보자고 제안했다.

판단:

- Scrapling은 흥미로운 프로젝트지만, 이번 이슈의 본질은 HTML 파싱보다
  "NID 로그인 흐름", "쿠키 복원", "에디터 iframe 준비", "Windows 배포 구조"에 가까웠다.
- 즉, 파서/스크래핑 계층을 바꿔도 현재 막히는 지점을 직접 해결하진 못한다.

결론:

- 이번 해결에는 도입하지 않았다.

#### 네이버 홈에서 메뉴를 따라가서 글쓰기 진입

사용자가 네이버 홈 우측 프로필 메뉴에서 블로그를 타고 들어가 글쓰기 버튼을 누르는 플로우도 제안했다.

판단:

- 이 방식은 사람이 보기에는 자연스럽지만, 자동화 입장에서는 클릭 단계가 늘어나서 오히려 변동 포인트가 많아진다.
- 현재 목표는 "한 번 로그인한 세션이 있으면 안정적으로 글쓰기 에디터에 바로 도달하는 것"이었다.

결론:

- 최종 검증 버전의 핵심 경로로는 채택하지 않았다.
- 다만 UI 구조가 크게 바뀌는 경우를 대비한 대안 경로 아이디어로는 가치가 있다.

#### 봇 탐지 역공학 수준의 깊은 로그인 위장

`cdc_*` 전역 변수, `navigator.webdriver`, Permissions API 등 더 깊은 스텔스 방향도 검토했다.

판단:

- 이건 장기적으로는 의미가 있지만, 이번 Windows 안정화의 즉시 해결 포인트는
  로그인 URL/세션 복원/에디터 대기 쪽이 더 결정적이었다.

결론:

- 현재 검증 완료 버전의 핵심 성공 요인은 깊은 스텔스보다 세션 재사용과 에디터 안정화였다.

## 5. 실제로 반영한 수정

아래는 복제본에 실제로 반영한 변경이다.

### 5.1 로그인 URL을 폼 모드로 고정

파일:

- `constants.py`

변경:

- `NAVER_LOGIN_URL = "https://nid.naver.com/nidlogin.login?mode=form"`

의미:

- QR 중심 기본 화면 대신 아이디/비밀번호 입력 폼이 보이는 로그인 페이지로 직접 진입하게 했다.
- 로그인 자동화가 출발선에서 막히지 않도록 한 가장 기초적인 수정이다.

### 5.2 세션 저장/복원 로직 강화

파일:

- `browser/session_manager.py`

변경:

- 쿠키 저장 시 기존 파일과 병합
- 세션 유효 기간을 24시간 수준이 아니라 30일로 확대
- 쿠키 복원 시 도메인별로 나눠서 순차 복원
  - `https://www.naver.com`
  - `https://blog.naver.com`
  - `https://section.blog.naver.com`

의미:

- 네이버 메인과 블로그 세션이 따로 놀지 않도록 했다.
- "한 번 로그인하면 계속 이어지게" 만들기 위한 핵심 변경이다.

### 5.3 에디터 준비 대기 강화

파일:

- `posting/editor.py`

변경:

- `WebDriverWait` 기반 `_wait_for_editor_ready()` 추가
- `ensure_editor_context()`가 기본 문서와 `mainFrame`, 후보 iframe들을 순차 확인
- 최대 20초까지 실제 Smart Editor 핵심 DOM이 뜰 때까지 대기
- 디버그 HTML 저장 보강

의미:

- iframe만 들어가면 성공이라고 보지 않고, 실제로 제목/툴바/본문 입력 가능한 DOM이 렌더링됐는지 확인하게 했다.
- "조기 포기" 문제를 줄였다.

### 5.4 글쓰기 재진입 시 NID 리다이렉트 대응 보강

파일:

- `posting/editor.py`
- `auth/naver_login.py`

변경:

- 글쓰기 URL 진입 시 다시 `nidlogin.login`으로 튀는 경우를 감지
- 현재 열린 NID 페이지에서 재로그인 시도
- 이후 블로그 세션 동기화를 다시 시도

의미:

- "메인 로그인은 됐는데 글쓰기 들어갈 때만 다시 로그인 요구"하는 케이스를 흡수했다.

### 5.5 발행 단계 안정화

파일:

- `posting/publisher.py`

변경:

- 누락된 `import time` 추가
- 발행/저장/확인 버튼 탐색은 텍스트와 셀렉터를 함께 보는 구조 유지

의미:

- `publish_now()` 경로가 Windows 검증 중 바로 깨지지 않도록 했다.

### 5.6 Windows 브라우저 실행 파일 탐지/프로필 관리 강화

파일:

- `browser/stealth_driver.py`
- `config.py`
- `config.yaml`
- `paths.py`

변경:

- Chrome/Chromium/Brave/Playwright Chromium 경로 후보를 Windows 기준으로 폭넓게 탐색
- `BROWSER_EXECUTABLE_PATH` 환경변수 또는 `config.yaml` 오버라이드 지원
- 브라우저 프로필을 `%APPDATA%\NaverBlogAuto\browser_profiles`에 저장
- `undetected_chromedriver` 재시도 시 `ChromeOptions`를 재사용하지 않고 새로 생성

의미:

- PC마다 브라우저 설치 위치가 달라도 동작 가능성을 높였다.
- 프로필/세션이 프로젝트 폴더가 아니라 사용자 데이터 영역에 쌓이도록 정리해 배포 친화적으로 만들었다.

### 5.7 Frozen 실행과 사용자 데이터 경로 정리

파일:

- `paths.py`

변경:

- 개발 실행과 PyInstaller/Nuitka 빌드 실행을 모두 고려한 `BUNDLE_DIR`, `APP_DATA_DIR` 구조 사용
- 세션, 디버그, 로그, 브라우저 프로필, 사용자 설정을 모두 `APP_DATA_DIR` 아래로 모음

의미:

- 다른 Windows PC로 옮겨도 현재 작업 폴더에 쓰기 권한이 없어서 깨지는 문제를 줄였다.

### 5.8 배포 스크립트와 배포 문서 정리

파일:

- `build.py`
- `README_배포.md`

변경:

- PyInstaller `onedir` 빌드 후 배포 문서를 자동 복사
- `dist\NaverBlogAuto-windows.zip` 자동 생성
- 배포 문서에 "exe 하나만 복사하면 안 된다"는 점을 명시
- SmartScreen/정책 차단/브라우저 경로 예외 설정 안내 추가

의미:

- 실제 배포 단위와 사용자 안내가 맞도록 정리했다.

## 6. 검증 과정

이번 작업은 "코드만 고쳐 놓고 끝"이 아니라, 단계별로 실제 실행 검증을 했다.

### 6.1 1차 스모크 테스트: 로그인 -> 에디터 진입

검증 항목:

- 브라우저 생성
- 세션 복원 또는 로그인
- 글쓰기 URL 이동
- 에디터 컨텍스트 확보

결과:

- 성공
- 최종적으로 `mainFrame` 기준 글쓰기 에디터까지 도달함

의미:

- 최소한 Windows에서 로그인/세션/글쓰기 진입은 이어졌다는 것을 확인했다.

### 6.2 2차 E2E 테스트: 임시저장 + 즉시 발행

테스트 흐름:

1. 로그인
2. 글쓰기 화면 진입
3. 제목 입력
4. 최소 본문 입력
5. 임시저장
6. `publish_now()` 실행
7. 발행 URL 확인
8. 모바일 블로그 페이지 HTTP 200 확인

실제 검증 결과:

- 테스트 제목: `[wincheck publish_now] 2026-04-22 03:12:38`
- 최종 URL: `https://blog.naver.com/likimartin/224260695163`
- 모바일 검증 URL: `https://m.blog.naver.com/likimartin/224260695163`
- 검증 응답: `200`

의미:

- 이번 작업의 핵심 목표였던 "Windows에서 로그인부터 발행까지 실제 완료"가 확인됐다.

### 6.3 3차 배포 테스트: exe 빌드

실행:

- `build.py`로 PyInstaller `onedir` 빌드

결과:

- 앱 폴더 생성: `dist\NaverBlogAuto`
- 배포 zip 생성: `dist\NaverBlogAuto-windows.zip`

의미:

- 개발 환경이 없는 사용자를 위한 배포물 생성까지 완료했다.

### 6.4 4차 Frozen 실행 테스트

확인 내용:

- 빌드된 `NaverBlogAuto.exe`가 실제로 기동 가능한지 확인
- `%APPDATA%\NaverBlogAuto` 아래에 세션/프로필/디버그 데이터가 생성되는지 확인

관찰:

- 한 차례는 실제 프로세스가 살아 있는 상태로 확인되어 번들 구조 자체는 정상임을 확인했다.
- 이후 재빌드된 exe 실행 시에는 로컬 Windows 애플리케이션 제어 정책이 서명되지 않은 exe를 차단했다.

해석:

- 이는 코드/의존성 누락 문제가 아니라 "서명되지 않은 배포물"에 대한 Windows 정책 문제다.
- 즉, 다른 개인용 PC에서는 실행될 수 있지만, 회사/학교 관리형 PC까지 100% 보장하려면 코드 서명이 필요하다.

## 7. 이번 작업에서 중요했던 판단

### 7.1 로그인 우회보다 세션 재사용이 더 중요했다

처음에는 CAPTCHA, 봇 탐지, 입력 방식이 가장 큰 문제처럼 보일 수 있었다.  
하지만 실제 Windows 안정화에서 가장 크게 먹힌 건

- 올바른 로그인 폼으로 들어가는 것
- 저장된 세션을 오래, 안전하게 다시 쓰는 것
- 메인/블로그 도메인 쿠키를 함께 관리하는 것

이었다.

즉, "매번 완벽한 자동 로그인"보다 "한 번 성공한 로그인을 오래 살리는 구조"가 훨씬 실용적이었다.

### 7.2 에디터 문제는 셀렉터 문제가 아니라 타이밍 문제였다

처음엔 셀렉터가 바뀐 것처럼 보였지만, 실제로는 iframe 전환 후 충분히 기다리지 않아 실패하는 케이스가 컸다.

즉,

- "못 찾는다"보다
- "너무 빨리 찾는다"

가 더 정확한 표현이었다.

### 7.3 배포는 exe 생성만으로 끝나지 않는다

Windows 배포에서 진짜 중요한 건

- 저장 경로 정리
- 브라우저 경로 다양성 대응
- `onedir` 구조를 이해한 배포 문서
- 서명 여부에 따른 정책 차단 리스크 인지

였다.

이걸 정리하지 않으면 "내 PC에선 되는데 남의 PC에선 안 되는" 상태가 계속된다.
