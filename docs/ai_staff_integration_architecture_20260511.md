# AIMAX 신규 AI 직원 통합 설계

작성일: 2026-05-11  
대상 서비스: `oracle/aimax-reports-api`

## 1. 배경

AIMAX에는 현재 예리, 현주 중심의 직원 카드와 작업 실행 흐름이 있다. 이번에 추가된 5명은 이미 별도 GitHub 레포에 화면/프로토타입 껍데기가 어느 정도 만들어져 있지만, 그대로 AIMAX에 합치는 방식은 위험하다. 각 레포는 기술 스택, 인증, 데이터 저장 방식, 실행 환경이 다르기 때문이다.

따라서 AIMAX는 "직원 사무실과 관리자 콘솔" 역할을 맡고, 각 직원 레포는 AIMAX 안에서 실행되는 기능 모듈 또는 AIMAX와 연결되는 외부 능력으로 흡수한다.

## 2. 핵심 원칙

1. 레포 전체를 한 번에 병합하지 않는다.
2. 직원 카탈로그, 관리자 설정, 작업 실행 계약을 먼저 만든다.
3. 사용자 화면에는 "이런 직원들이 생긴다"는 예고를 보여주되, 실행 가능한 직원과 준비 중인 직원을 명확히 구분한다.
4. 관리자 페이지에서 직원 노출, 상태, 이미지, 권한, 연결 방식, 설정 누락 여부를 관리할 수 있게 한다.
5. AI 호출, 회계 데이터, 로컬 앱 연결처럼 비용/보안/운영 위험이 큰 기능은 별도 단계로 격리한다.

## 4. 통합 타입 정의

`planned`
: 프로필과 설명만 노출한다. 실행 기능은 없다.

`web_module`
: AIMAX 웹 안에 직원별 작업 화면을 제공한다. 별도 레포의 화면을 참고하되, AIMAX 인증/권한/레이아웃에 맞게 재구성한다.

`server_ai`
: AIMAX 서버에서 AI 작업을 실행하고, 결과/로그/비용/에러를 AIMAX 데이터에 남긴다.

`server_data`
: 서버가 데이터 저장, 조회, 집계, 내보내기, 감사 로그를 책임진다. 회계/프로젝트 관리처럼 데이터 정합성이 중요한 직원에 사용한다.

`local_agent`
: 기존 예리/현주처럼 사용자 PC의 로컬 에이전트와 연결되어 실행된다.

`local_app`
: 데스크톱 앱 또는 별도 설치 앱으로 제공한다. AIMAX는 다운로드, 버전, 연결 상태, 실행 안내, 결과 수신만 담당한다.

`external`
: 나중에 별도 배포된 외부 서비스와 API로 연결할 때 사용한다.

## 5. 직원별 통합 전략

### 윤미: 스크립트작가

윤미 레포는 이미 AI 분석/생성 서버 함수와 결과 스키마가 있어 가장 먼저 실제 기능화하기 좋다. 단, 현재 레포의 인증, Supabase, Lovable AI Gateway 구조를 그대로 가져오면 AIMAX 운영 모델과 비용 통제가 흐려진다.

권장 방식:
- UI는 AIMAX 직원 모듈 화면으로 재구성한다.
- `analyzeReference`, `generateContent`의 도메인 로직과 출력 스키마를 AIMAX 서버 작업으로 이식한다.
- 작업은 `staff_jobs`에 저장하고 `staff_code = "yunmi"`로 추적한다.
- AI 모델, 비용 라벨, 자동 재시도 금지, 기존 요청 ID 복구를 기본 정책으로 둔다.
- 첫 버전은 "레퍼런스 분석 -> A/B/C 스크립트 생성 -> 결과 저장"까지만 목표로 한다.

### 송이: 자료조사원

송이 레포는 콘텐츠 리서치 UI 껍데기가 잘 잡혀 있다. 아직 실제 수집/분석은 목 데이터에 가깝다.

권장 방식:
- 먼저 AIMAX 안에 리서치 보드형 `web_module`로 붙인다.
- URL, 플랫폼, 카테고리, 태그, 분석 요약을 저장할 `references` 또는 `research_items` 모델을 만든다.
- 실제 크롤링/수집은 후순위로 두고, 초기에는 사용자가 직접 넣은 레퍼런스를 정리/분석하는 기능부터 붙인다.
- 윤미와 연결해 "송이가 찾은 자료 -> 윤미가 스크립트 생성" 흐름을 만들 수 있게 데이터 구조를 맞춘다.

### 현성: PM

현성 레포는 프로젝트/태스크/일정 변경 이유 관리 프로토타입이 있다. AIMAX에서 직원과 작업이 늘어날수록 운영 대시보드와 연결 가치가 커진다.

권장 방식:
- 직원별 작업, 요청, 상태, 마감일을 묶는 내부 운영 모듈로 시작한다.
- 기존 브라우저 localStorage 구조는 서버 저장으로 바꾼다.
- 관리자 페이지의 직원 관리와 연결해 "직원별 설정 누락", "최근 실패", "대기 작업"을 볼 수 있게 한다.
- 추후 사용자별 프로젝트 관리로 확장하되, 첫 단계는 운영자용 내부 PM 기능에 가깝게 둔다.

### 상수: 회계원

상수 레포는 Supabase 기반 회계 대시보드와 매출/비용/상품/채널 데이터 구조가 있다. 실사용 가치가 높지만, 개인정보와 재무 데이터가 들어갈 수 있어 가장 조심해야 한다.

권장 방식:
- 화면 껍데기는 참고하되, AIMAX 데이터 저장 정책이 먼저 필요하다.
- `sales`, `expenses`, `products`, `channels` 같은 도메인 모델은 유지 가능하다.
- 권한, 백업, CSV 내보내기, 감사 로그, 삭제/수정 이력 없이는 실사용 배포하지 않는다.
- 첫 버전은 샘플 데이터 또는 관리자 전용 베타로 제한한다.

### 나경: 판서쌤

나경 레포는 Electron/Fabric.js 기반 데스크톱 판서 앱이다. 웹 모듈로 억지 편입하기보다 별도 로컬 앱으로 취급해야 한다.

권장 방식:
- AIMAX에는 직원 카드, 설치 상태, 최신 버전, 다운로드 링크, 연결 상태만 둔다.
- 데스크톱 앱은 별도 빌드/배포 파이프라인을 갖는다.
- 나중에 AIMAX 계정과 페어링하거나 결과 파일/녹화/이미지를 AIMAX로 업로드하는 구조를 만든다.
- 초기에는 `local_app` 타입의 준비 중 직원으로 관리한다.

## 6. 직원 카탈로그 모델

현재는 서버 코드 안의 `WORKERS` 배열이 직원 정보를 직접 가진다. 앞으로는 데이터 기반으로 분리해야 관리자 페이지에서 관리할 수 있다.

권장 저장소:
- 단기: `DATA_DIR/staff-catalog.json`
- 중기: SQLite 또는 기존 서비스 데이터 저장소
- 장기: 권한/작업/감사 로그와 함께 정규화된 DB

권장 필드:

```json
{
  "code": "yunmi_script_writer",
  "staff_code": "yunmi",
  "name": "윤미",
  "role": "스크립트작가",
  "title": "숏폼/블로그 스크립트 설계",
  "description": "레퍼런스를 분석하고 A/B/C 스크립트 초안을 만듭니다.",
  "category": "content",
  "product": "bundle",
  "status": "needs_setup",
  "visibility": "visible",
  "sort_order": 40,
  "integration_type": "server_ai",
  "module_key": "script_writer",
  "job_kind": "",
  "execution": {
    "type": "planned",
    "runner": null
  },
  "required_settings": ["future_staff_setup"],
  "capabilities": ["reference_analysis", "script_generation"],
  "profile_image": "/assets/avatar_yunmi.jpg",
  "repo_url": "https://github.com/makefamily/script-writer",
  "source_commit": "29436b4de9cb38279de3f922dca1e7ccadfaa037",
  "admin_notes": "",
  "created_at": "2026-05-11T00:00:00.000Z",
  "updated_at": "2026-05-11T00:00:00.000Z"
}
```

## 7. 상태와 노출 정책

권장 `status`:
- `planned`: 등록 예정
- `needs_setup`: 설정 필요
- `beta`: 관리자 또는 일부 사용자에게만 제공
- `available`: 일반 사용 가능
- `disabled`: 숨김 또는 비활성

권장 `visibility`:
- `hidden`: 사용자 화면에 표시하지 않음
- `visible`: 직원 카드에는 표시
- `admin_only`: 관리자 화면에만 표시

실행 가능 여부는 `status`만으로 판단하지 않는다. 아래 조건을 모두 만족해야 한다.

1. 사용자가 해당 상품 또는 직원 권한을 가진다.
2. 직원이 `visible` 또는 사용자의 베타 대상이다.
3. `integration_type`에 맞는 실행기 또는 모듈 설정이 있다.
4. 필수 설정이 모두 채워져 있다.

## 8. API 설계

기존 사용자 API:
- `GET /api/workers`
  - 계속 유지한다.
  - 민감한 관리자 필드는 제외한다.
  - 기존 프론트가 깨지지 않도록 `code`, `name`, `role`, `avatar`, `settings`, `product`, `status`, `repo_url`은 유지한다.

관리자 API:
- `GET /api/admin/staff`
- `GET /api/admin/staff/:code`
- `POST /api/admin/staff`
- `PATCH /api/admin/staff/:code`
- `POST /api/admin/staff/:code/rollout`
- `POST /api/admin/staff/:code/avatar`
- `GET /api/admin/staff/:code/jobs`
- `GET /api/admin/staff/:code/audit`

작업 API:
- 기존 `/api/jobs`는 예리/현주 호환을 위해 유지한다.
- 신규 직원은 `POST /api/staff-jobs`를 권장한다.
- 공통 필드:

```json
{
  "staff_code": "yunmi",
  "module_key": "script_writer",
  "action": "generate_script",
  "input": {},
  "request_id": "client-generated-id",
  "cost_acknowledged": true
}
```

## 9. 관리자 페이지 설계

관리자 페이지에는 "직원 관리" 섹션을 추가한다.

필수 화면:
- 직원 목록
- 상태 필터: 전체, 사용 가능, 설정 필요, 베타, 숨김
- 직원 상세/편집 패널
- 이미지, 이름, 역할, 설명, 정렬 순서 수정
- 상품/권한 연결
- 통합 타입과 모듈 키 수정
- 레포 URL과 소스 커밋 기록
- 필수 설정 체크리스트
- 최근 작업/오류/비용 요약

첫 구현에서 꼭 필요한 기능:
- 직원 목록 조회
- 직원 상태 변경
- 노출 여부 변경
- 정렬 순서 변경
- 설명/이미지/레포 URL 수정

후순위:
- 이미지 업로드
- 베타 대상 사용자 지정
- 작업 로그 상세
- 감사 로그 UI

## 10. 사용자 화면 설계

사용자 화면은 카탈로그를 기준으로 그린다.

원칙:
- 준비 중 직원은 카드로 보여준다.
- 준비 중 직원은 작업 선택 목록과 빠른 실행 버튼에서 제외한다.
- 카드에는 `설정 필요`, `준비 중`, `베타` 같은 명확한 상태 배지를 표시한다.
- 기능이 붙은 직원은 직원별 작업 화면으로 진입시킨다.

## 11. 단계별 추진안

### S0. 현재 상태 고정

목적:
- 이미 배포된 5명 카드가 깨지지 않게 유지한다.

산출물:
- 현재 `/api/workers` 응답 구조 기록
- 직원별 이미지 경로와 레포 URL 기록

검증:
- `/api/workers`에서 기존 2명 + 신규 5명 조회
- 신규 직원은 실행 목록에서 제외

### S1. 데이터 기반 직원 카탈로그

목적:
- 코드 하드코딩을 줄이고 관리자 페이지 기반으로 이동할 바탕을 만든다.

산출물:
- `staff-catalog.json`
- `loadStaffCatalog()`, `saveStaffCatalog()`
- 기본 seed 로직

검증:
- 파일이 없으면 현재 직원 목록으로 자동 생성
- 파일이 있으면 서버 재시작 후 변경 사항 유지

### S2. 관리자 직원 관리

목적:
- 운영자가 신규 직원을 코드 수정 없이 관리한다.

산출물:
- 관리자 API
- 관리자 페이지 직원 관리 탭

검증:
- 관리자만 접근 가능
- 상태/노출/정렬/설명 수정 후 `/api/workers` 반영

### S3. 공통 작업 계약

목적:
- 직원별 기능을 같은 방식으로 실행, 추적, 실패 복구한다.

산출물:
- `staff_jobs` 저장 구조
- `POST /api/staff-jobs`
- 작업 상태, 입력, 결과, 에러, 비용 필드

검증:
- 실행기가 없는 직원은 명확한 `needs_setup` 응답
- 기존 예리/현주 작업 흐름은 유지

### S4. 윤미 1차 기능화

목적:
- 가장 완성도가 높은 AI 직원부터 실제 사용 가능한 기능으로 만든다.

산출물:
- 레퍼런스 분석
- 스크립트 생성
- 결과 저장/조회

검증:
- 유료 AI 호출 전 명시적 비용 확인
- 실패 시 기존 요청 ID로 복구 가능
- 자동 재시도 없음

### S5. 송이/현성/상수 모듈화

목적:
- 각 레포의 화면 껍데기를 AIMAX 운영 모델에 맞게 흡수한다.

산출물:
- 송이 리서치 보드
- 현성 PM 보드
- 상수 회계 베타 대시보드

검증:
- AIMAX 인증/권한 사용
- 데이터가 서버에 저장
- 관리자에서 노출/권한 제어 가능

### S6. 나경 로컬 앱 연결

목적:
- Electron 판서 앱을 AIMAX 직원으로 관리한다.

산출물:
- 다운로드/버전 정보
- 연결 상태
- 결과 파일 업로드 또는 페어링 설계

검증:
- 웹 배포와 데스크톱 앱 배포가 분리됨
- 관리자에서 버전/링크 변경 가능

## 12. 보안과 비용 안전

AI 직원 공통:
- 유료 호출은 기본적으로 자동 실행하지 않는다.
- 사용자가 비용 발생을 명시적으로 확인해야 한다.
- 실패 후 재시도 전 기존 요청 ID를 먼저 조회한다.
- API 키는 서버 환경 변수에만 둔다.
- 클라이언트에는 모델 키, signed URL, 내부 오류 스택을 노출하지 않는다.

회계 직원:
- 관리자 전용 베타부터 시작한다.
- 수정/삭제 감사 로그를 남긴다.
- CSV 내보내기와 백업 정책을 먼저 둔다.

로컬 앱:
- 앱 다운로드 링크와 버전 메타데이터를 분리 관리한다.
- 앱에서 AIMAX로 보내는 데이터는 사용자 승인 기반으로 한다.

## 13. 다음 AI 개발자에게 주는 결론

첫 구현의 목표는 "직원 기능 구현"이 아니다. 목표는 "직원을 운영 가능한 단위로 관리하는 기반"이다.

가장 먼저 해야 할 일:
1. 하드코딩된 직원 목록을 데이터 기반 카탈로그로 분리한다.
2. 관리자 페이지에서 직원 상태와 노출을 관리하게 한다.
3. 실행 가능한 직원과 준비 중 직원을 같은 카탈로그에서 구분한다.
4. 이후 윤미부터 공통 작업 계약에 태운다.
